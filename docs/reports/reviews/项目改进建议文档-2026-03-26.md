# 项目改进建议文档

审查时间：2026-03-26
审查范围：后端路由层、项目状态管理、写盘路径、前端核心模块、前端测试分布
审查方式：静态代码审查 + 现有测试与目录结构核对

---

## 一、结论

当前项目功能面已经比较完整，近期用户可见问题（Copilot 命令区纯文字化、工作流面板去路径化）也已经被修正并有回归测试保护。
但从工程维护角度看，项目仍有 8 个值得继续推进的改进点，其中最优先的不是“再加功能”，而是收敛运行态同步、减少重复编排逻辑、统一关键写盘路径、降低前端全局耦合。

如果只排一轮优先级，建议顺序如下：

1. **P1：统一配置刷新与项目切换后的运行态同步**
2. **P1：收敛 Router / Chat / Continuous Write 中的重复编排逻辑**
3. **P1：统一关键数据写入为原子写入**
4. **P2：治理前端全局挂载与脚本顺序耦合**
5. **P2：重构 aux-memory 模块并补测试**
6. **P3：规划 `/api` 与 `/api/v1` 双前缀收敛路径**

---

## 二、高优先级问题（P1）

### 1. 配置保存成功，但运行态刷新入口仍然分散

**问题**

`settings` 路由已经在保存和重载后显式重建 `NovelCoordinator`，并同步 `RouterAgent`；这说明项目已经意识到“写入配置文件”和“刷新内存运行态”不是一回事。
但目前这套逻辑仍然散落在具体路由里，没有看到一个统一的运行态刷新入口。

**证据**

- `novel_agent/web/routes/settings.py:38-42`
- `novel_agent/web/routes/settings.py:94-118`
- `novel_agent/web/routes/settings.py:123-142`

核心表现：

1. `_sync_router_coordinator()` 专门负责同步 RouterAgent 的 coordinator 引用。
2. `/settings` 保存后会 `Config.reload()`，随后重建 `NovelCoordinator`。
3. `/settings/reload` 也会重复同样的重建逻辑。

**影响**

这类逻辑如果只在部分入口维护，会带来两个后果：

1. 以后新增配置入口时容易漏掉运行态刷新。
2. 某些改动“磁盘上已生效、内存里未生效”，最终表现成偶发错乱。

**建议**

建议抽出统一的 runtime refresh 服务，例如：

1. 统一执行 `Config.reload()`。
2. 统一重建 coordinator。
3. 统一同步 router / websocket / 依赖容器。
4. 所有配置变更入口只调用这一处。

---

### 2. 项目切换只切了 project id，运行上下文刷新不完整

**问题**

当前项目切换 API 只调用 `pm.switch_project(project_id)`，而 `ProjectManager.switch_project()` 只更新 `current_project_id` 并保存元数据，没有看到与聊天会话、workflow、知识库、writer 缓存、router/coordinator 上下文的统一同步。

**证据**

- `novel_agent/web/routes/projects.py:470-476`
- `novel_agent/project_manager.py:158-163`

**影响**

这会导致跨项目场景存在潜在上下文漂移：

1. 已存在的 chat session 可能还挂在旧项目上下文上。
2. workflow 快照可能还保留旧项目状态。
3. continuous writer / knowledge base 这类 project-scoped 资源可能继续使用旧项目数据。

**建议**

建议为“项目切换”建立统一的激活流程：

1. 切换 `current_project_id`。
2. 清理或隔离当前 chat / workflow / writer 缓存。
3. 重建或重绑项目相关上下文管理器。
4. 返回明确的“已切换并完成运行态同步”结果。

### 3. RouterAgent 与 Coordinator 私有实现耦合过深

**问题**

`RouterAgent` 不只是做意图编排，还直接依赖 `Coordinator` 的私有方法，并承担 outline / chapter / compiled novel 的持久化职责。

**证据**

- `novel_agent/agents/router_agent.py:999-1000`
- `novel_agent/agents/router_agent.py:1028-1037`
- `novel_agent/agents/router_agent.py:1096-1145`
- `novel_agent/agents/router_agent.py:1177-1187`

具体表现：

1. `self.coordinator._extract_chapters(...)`
2. `pm.save_project_data("outline", ...)`
3. `chapter_file.write_text(...)`
4. `self.coordinator._save_novel(...)`

**影响**

这会带来明显的边界污染：

1. RouterAgent 改动会牵动 coordinator 内部实现。
2. coordinator 私有方法一旦调整，router 很容易跟着坏。
3. 编排层同时负责“决策 + 文件写盘 + 数据转换”，可测试性下降。

**建议**

建议把这部分收敛成公开服务层：

1. 由独立的 project content service 负责 outline/chapter/novel 持久化。
2. coordinator 暴露稳定的 public API，而不是让 router 调私有方法。
3. router 只组装上下文和选择执行路径，不直接碰私有落盘细节。

---

### 4. 关键写盘路径未统一使用原子写入

**问题**

项目里已经有 `atomic_write_json` / `atomic_write_text` 基建，但关键路径仍存在直接 `write_text()`。

**证据**

- `novel_agent/project_manager.py:89-95` 已使用 `atomic_write_json(...)`
- `novel_agent/project_manager.py:242-251` 的 `save_project_data()` 仍直接 `path.write_text(...)`
- `novel_agent/project_manager.py:126-129` 项目初始化时多个 JSON 文件直接 `write_text(...)`
- `novel_agent/agents/router_agent.py:1133-1138` 章节 markdown 直接 `write_text(...)`

**影响**

在应用运行中断、异常退出或并发写入时，直接写盘更容易出现：

1. 文件被截断
2. JSON 半写入
3. 章节内容损坏
4. 前端读取到不完整状态

**建议**

建议做一次“写盘路径统一治理”：

1. 项目元数据、outline、章节正文、导入结果全部统一走 atomic write。
2. 明确哪些文件允许直接覆盖，哪些必须保留旧内容用于回滚。
3. 把“是否需要原子写入”从经验判断变成规范。

---

### 5. `/chat` 与 `/chat/stream` 存在大段平行复制逻辑

**问题**

普通聊天与 SSE 流式聊天在进入执行前，做了非常相似的一整套流程：

1. session 归一化
2. project/session key 计算
3. 安全检查
4. explicit / targeted command 解析
5. workflow control 处理
6. session 恢复
7. model refresh
8. routing hint 分析
9. router 执行判断

**证据**

- `novel_agent/web/routes/chat.py:1155-1357`
- `novel_agent/web/routes/chat.py:1378-1661`

从代码结构上看，这两条路由几乎是“同一套编排，输出通道不同”。

**影响**

重复逻辑的风险很直接：

1. 修一个接口漏另一个。
2. 普通接口和流式接口行为漂移。
3. 每次增加 routing / workflow 字段都要改两份。

**建议**

建议抽出共享 orchestration 层，例如：

1. `prepare_chat_request(...)`
2. `resolve_or_restore_chat_session(...)`
3. `build_routing_hint(...)`
4. `execute_router_flow(...)`
5. `persist_chat_session(...)`

这样 `/chat` 与 `/chat/stream` 只保留“返回 JSON”与“返回 SSE”的差异。

---

### 6. continuous_write 中 API 配置装配逻辑重复

**问题**

`continuous_write` 的 start / continue / regenerate 都在手动拼装同一套配置回退逻辑：

1. 优先读指定 `api_config_id`
2. 不足时回退到全局配置
3. 再单独解析 model
4. 再手动构建 `AgentModelConfig`

**证据**

- `novel_agent/web/routes/continuous_write.py:313-334`
- `novel_agent/web/routes/continuous_write.py:470-507`
- `novel_agent/web/routes/continuous_write.py:878-915`

**影响**

这类重复逻辑非常容易出现行为漂移：

1. 某处忘记 cap max_tokens。
2. 某处忘记同步默认 model。
3. 某处修了 fallback，另一处还保留旧逻辑。

**建议**

建议提取成一个统一解析函数，例如：

1. 输入：`request.model`、`request.api_config_id`
2. 输出：标准化后的 `AgentModelConfig` 或配置对象
3. 内部统一处理 fallback / 默认模型 / token cap

## 三、中优先级问题（P2）

### 7. 前端核心模块仍高度依赖全局挂载与脚本顺序

**问题**

前端并不是模块化打包体系，而是大量脚本通过全局 `window` 互相协作。

**证据**

- `novel_agent/web/static/app-core.js:1147-1166`

这里直接把大量核心能力挂到全局：

1. `window.store = store`
2. `window.ui = ui`
3. `window.init = init`
4. `window.updateCopilotWorkflowPanel = ...`
5. `window.restoreCopilotWorkflowStatus = ...`

**影响**

这种方式短期灵活，但长期会增加：

1. 脚本加载顺序依赖
2. 隐式耦合
3. 模块边界不清
4. 测试初始化成本高

**建议**

建议逐步收敛，而不是一次重写：

1. 先收敛成少量命名空间对象，而不是散落的全局函数。
2. 新增模块优先依赖显式注入，不再继续扩大 `window.*`。
3. 将最核心的共享状态访问统一到一个 app shell 接口层。

---

### 8. Copilot 初始化逻辑有重复绑定风险

**问题**

`initCopilotEnhancements()` 每次执行都会给同一个输入框重复绑定 `input` / `keydown` / `blur` 监听器，没有看到显式的防重复初始化保护。

**证据**

- `novel_agent/web/static/app-copilot.js:457-471`

**影响**

如果页面切换、局部重挂载、二次初始化路径再次触发这段逻辑，可能出现：

1. 事件重复触发
2. 自动补全弹层行为异常
3. helper 重复渲染
4. 难以复现的输入问题

**建议**

建议增加最小防护：

1. 在元素上打初始化标记。
2. 或使用单次绑定策略。
3. 至少让初始化函数具备幂等性。

---

### 9. aux-memory 模块可维护性明显偏弱，且缺少专门前端回归测试

**问题**

`app-aux-memory.js` 目前是前端里最典型的“大段 innerHTML + 内联样式 + 全量重渲染 + 再全量绑事件”模块。

**证据**

- `novel_agent/web/static/app-aux-memory.js:110-147`
- `novel_agent/web/static/app-aux-memory.js:162-267`
- `novel_agent/web/static/app-aux-memory.js:284-373`

表现包括：

1. `innerHTML` 大片段拼接
2. `style.cssText` 内联布局与颜色
3. 每次渲染后重新全量绑定事件
4. UI、状态、交互、模板强耦合在同文件

同时，当前 `frontend-tests` 目录里只有：

- `settings-short-story.dom.test.js`
- `copilot-workflow.dom.test.js`
- `continuous-write.dom.test.js`
- `novel-to-script.dom.test.js`
- `copilot-commands.dom.test.js`

没有看到 aux-memory 对应的 DOM 回归文件。

**影响**

这会导致：

1. 改动成本高
2. 样式一致性难维护
3. 事件重复绑定风险高
4. 真实回归没有测试兜底

**建议**

建议把它作为一个单独治理专题：

1. 先提取 class 和结构模板，减少 `style=""`。
2. 再把分类列表、条目列表、详情编辑区拆成几个渲染函数。
3. 最后补 2~3 条 DOM 测试，至少覆盖筛选、选择、批量操作、编辑保存。

## 四、低优先级问题（P3）

### 10. `/api` 与 `/api/v1` 双前缀并存，兼容价值存在，但维护成本也在累积

**问题**

项目当前同时挂载了版本化和无版本前缀两套路由。

**证据**

- `novel_agent/web/routes/__init__.py:70-75`

**影响**

短期看这能保兼容；长期看会增加：

1. 路由文档维护成本
2. 测试矩阵成本
3. 前端和第三方接入迁移不清晰的问题

**建议**

不建议立即删除无版本前缀，但建议开始做收敛计划：

1. 明确对外推荐使用 `/api/v1`。
2. 统计前端与外部调用仍依赖 `/api` 的位置。
3. 逐步迁移后，再决定是否保留兼容层。

---

## 五、建议的实施顺序

### 第一轮（最值得马上做）

1. 统一配置刷新与项目切换运行态同步。
2. 抽出 chat / chat-stream 共用编排层。
3. 把 outline / chapter / project data 写盘统一为 atomic write。

### 第二轮（降低维护成本）

1. 抽取 continuous_write 配置解析函数。
2. 给 Copilot 初始化增加幂等保护。
3. 开始收敛前端全局挂载边界。

### 第三轮（前端治理与长期演进）

1. 重构 aux-memory。
2. 为 aux-memory 补 DOM 测试。
3. 规划 API 前缀迁移策略。

---

## 六、总体判断

这个项目当前最需要的不是“大改架构”，而是把已经出现的几个重复模式和隐式耦合点收口。
从收益看，**运行态同步统一、chat 编排收敛、原子写入统一** 这三件事的性价比最高：

1. 能直接降低线上错乱概率。
2. 能减少后续功能开发时的重复改动。
3. 能给前端和路由层继续迭代提供更稳定的基础。

如果后续要继续做下一轮，我建议先从这三项开始，而不是继续扩展新功能。
