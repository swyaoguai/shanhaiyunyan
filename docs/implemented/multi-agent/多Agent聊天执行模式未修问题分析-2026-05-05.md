# 多 Agent 聊天执行模式未修问题分析

## Related Files

- `novel_agent/web/routes/chat.py` - Copilot 聊天入口、流式聊天、工作流状态与 active run 管理。
- `novel_agent/web/static/app-core.js` - Copilot 前端发送、停止、SSE 处理、合同确认卡片与工作流面板。
- `novel_agent/agents/communicator.py` - 普通对话的信息收集与流式回复实现。
- `novel_agent/agents/router_agent.py` - 意图路由、合同草案、正式多助手协作执行链入口。
- `novel_agent/workflow/project_ready.py` - 正式任务池 ready-task 执行循环。
- `novel_agent/workflow/coordinator.py` - 合同初始化、任务池持久化、项目产物落盘与协调器状态。
- `novel_agent/workflow/contracts.py` - 创作合同与默认任务图。
- `novel_agent/web/routes/novel.py` - 合同确认 API 与首批任务执行。
- `docs/监督式自组织多Agent演进方案-2026-03-27-实施进度.md` - 已落地能力的历史说明，用于对照现状。

## 用户目标

用户期望的多 agent 模式不是简单改变“生成/停止”按钮状态，而是让聊天讨论本身成为创作决策与内容修改入口：

1. 用户可以在同一个聊天框中持续和助手讨论。
2. 助手要根据聊天内容决定创作流程、创作方向、当前应该继续讨论还是进入执行。
3. 用户在聊天中提出的更正，应能修改所有可被创作的内容，包括世界观、角色、事件线、大纲、细纲、章纲、正文、润色稿、阶段总结等。
4. 系统应能把讨论沉淀为可执行状态：更新合同、任务池、项目资料、章节内容或后续生成约束。
5. 已有内容和待生成内容都应被统一纳入“可修改对象”，而不是只支持新建一条固定创作流程。

当前代码已经有合同、任务池、协作轨迹、多个创作 Agent、工作流面板等基础设施，但距离上述交互模型仍有几个关键缺口。

## 结论

当前实现更接近“聊天中触发一段固定创作流程”，还不是“由聊天持续驱动创作决策、创作方向和全量内容修改的多 agent 工作台”。

最影响用户体验的未修问题有三个：

1. 前端主路径走 `/api/chat/stream`，但流式聊天不会结构化更新 `collected_info`。
2. 聊天讨论没有统一转换成“创作决策 / 修改指令 / 任务池变更”，反馈大多仍停留在普通对话层。
3. 已有项目内容没有形成统一可编辑对象模型，无法稳定覆盖“所有可被创作的内容”的修改。

## P0-1：流式聊天不会更新已收集信息

### 现象

前端 Copilot 默认调用 `/api/chat/stream`。普通非流式 `chat()` 会解析 LLM 返回并执行：

- `novel_agent/agents/communicator.py:353` - `self.collected_info.update(result["extracted_info"])`

但流式 `chat_stream()` 只生成自然语言回复，并在 done 事件里原样返回当前 `self.collected_info`：

- `novel_agent/agents/communicator.py:404` - `async def chat_stream(...)`
- `novel_agent/agents/communicator.py:484` - 构造 `done_data`
- `novel_agent/agents/communicator.py:488` - 返回 `"collected_info": self.collected_info`

流式路径没有等价的 `extracted_info` 解析与写回。

### 影响

用户前面通过聊天给出的类型、主角、风格、禁忌、世界观修正等信息，可能只留在自然语言历史里，没有进入 `collected_info`。后续 Router 构建创作合同时依赖 `agent.collected_info`：

- `novel_agent/web/routes/chat.py:1016` 附近构造 router context
- `novel_agent/web/routes/chat.py:1032` 把 `conversation_history` 放进去，但结构化字段仍来自 `collected_info`

这会导致“聊了很多，执行时没带上关键信息”。

### 为什么还没满足目标

用户要求“边聊天边收集信息更正信息”。当前主路径只是边聊天边保留自然语言历史，不是可靠的结构化收集与修正。

## P0-2：聊天讨论没有成为统一的创作决策与修改入口

### 现象

当前聊天入口会先做意图判断，并在特定触发词或模式下进入 Router 执行链：

- `novel_agent/web/routes/chat.py:971` 附近 - `_should_execute_via_router`
- `novel_agent/web/routes/chat.py:987` 附近 - `_infer_communicator_response_mode`
- `novel_agent/web/routes/chat.py:1014` 附近 - `_build_router_context`
- `novel_agent/agents/router_agent.py:1123` 附近 - `route_and_respond`

但这个链路的核心仍是“识别意图 → 分发到固定 Agent → 返回结果”。它没有一个统一的“创作决策层”来处理以下聊天语义：

- 继续讨论，不落盘。
- 根据讨论更新创作方向。
- 修改已生成世界观。
- 修改已有角色档案。
- 修改事件线、大纲、细纲、章纲。
- 修改某一章正文或某一段内容。
- 修改后续生成约束，但不改已有文件。
- 修改已有文件，并同步后续任务。

Router 有 `provide_feedback` 意图，但聊天路由中它被映射到普通 `Communicator`：

- `novel_agent/web/routes/chat.py:40` - `"provide_feedback": "Communicator"`

代码里虽然存在 `REPLAN_REQUIRED` 等消息协议，但聊天路径没有把用户反馈转成“修改合同 / 修改任务池 / 修改项目数据 / 修改章节文件”的统一闭环。

### 影响

用户说“刚才那个主角设定改成女性，不要修仙，改成赛博都市”，系统可能只生成一段对话回复，或者只影响后续自然语言上下文。它不会稳定地判断应该修改哪些对象：

- `creation_contract`
- `task_pool`
- `worldbuilding.json`
- `characters.json`
- `eventlines.json`
- `outline.json`
- `detail_settings.json`
- `chapter_settings.json`
- `chapters/*.md`
- 阶段总结或合集文件

结果就是：聊天讨论对创作方向有影响，但影响路径不稳定、不可验证，也不一定会落到已有内容。

### 为什么还没满足目标

目标不是“执行时还能不能按发送按钮”，而是“聊天内容能不能决定创作流程、创作方向、以及对全部可创作内容的修改”。当前系统缺少这个统一决策与修改层。

## P0-3：缺少“所有可创作内容”的统一可编辑对象模型

### 现象

当前不同内容类型分散在不同模块、不同项目数据文件和不同生成链路中：

- 世界观由世界观构建链路和 `worldbuilding.json` 承载。
- 角色由 `CharacterBuilder`、`CharacterManager`、`characters.json` 承载。
- 大纲由 `Outliner`、`outline.json` 承载。
- 事件线、细纲、章纲由各自 builder 和项目资料承载。
- 正文由章节写作链路、`outline` 行内内容和 `chapters/*.md` 承载。
- 阶段总结由 `summary_orchestrate` 和 `stage_summaries/*.md` 承载。

但是聊天层没有统一的对象索引，例如：

- 用户说“改一下第二卷主线”时，对应哪些文件、哪些任务、哪些后续约束？
- 用户说“这个角色动机不对”时，是改角色档案、已写章节、后续章纲，还是全部都改？
- 用户说“把前面伏笔删掉”时，应查找哪些章节、事件线和总结？
- 用户说“以后不要走复仇线”时，是仅更新后续生成约束，还是重写已有大纲？

### 影响

没有统一可编辑对象模型时，助手只能依赖意图关键词和局部逻辑。它难以保证“所有可被创作的内容”都能被发现、选择、修改、回写、同步后续任务。

### 为什么还没满足目标

用户要求的是全域创作修改能力，不只是创建新内容。当前缺少“聊天请求 → 目标对象识别 → 修改策略 → 落盘 → 后续任务同步”的通用链路。

## P1-1：合同确认绕过聊天工作流，确认后只执行首批任务

### 现象

前端合同确认按钮直接调用：

- `novel_agent/web/static/app-core.js:2266` - 读取 `response.task_pool`
- `novel_agent/web/static/app-core.js:2280` 附近点击后只 append “合同已确认，正式任务池已初始化。”

后端 `/contract/confirm` 直接初始化任务池并执行首批：

- `novel_agent/web/routes/novel.py:336` - `@router.post("/contract/confirm")`
- `novel_agent/web/routes/novel.py:367` - `initialize_task_pool_from_contract`
- `novel_agent/web/routes/novel.py:368` - `execute_project_ready_tasks`
- `novel_agent/web/routes/novel.py:369` - `max_tasks=4`
- `novel_agent/web/routes/novel.py:370` - `max_chapter_tasks=2`

### 影响

合同确认不是“启动一个可持续的后台 run”。它执行一个有限 batch 后结束。后续继续执行、用户中途修正、根据反馈重规划，都没有形成统一聊天闭环。

### 为什么还没满足目标

确认动作和聊天 run 是两套路径。用户在聊天框里讨论出的合同，确认后进入另一个 API 流程，交互连续性不足。

## P1-2：正式任务池是串行调度，不是真正多 agent 并行协作

### 现象

`ProjectReadyTaskExecutor` 每轮只拿第一个 ready task：

- `novel_agent/workflow/project_ready.py:421` - `current_task = ready_tasks[0]`

循环上限控制为 batch 数：

- `novel_agent/workflow/project_ready.py:413` - `max_task_limit`
- `novel_agent/workflow/project_ready.py:414` - `chapter_task_limit`
- `novel_agent/workflow/project_ready.py:507` - `max_tasks_reached`

### 影响

虽然界面和文档称为多 agent，但正式任务池执行仍是 Coordinator 按顺序调用固定 handler。它缺少真正的多 worker 并行认领、冲突合并、用户修正事件广播、运行中重新规划等能力。

### 为什么还没满足目标

用户看到“多 agent 模式”时，预期可能是多个助手分别收集资料、写文件、修正上下文并持续汇报。当前只是串行编排多个 Agent。

## P1-3：active run 仍是单槽状态，难以承载连续讨论后的多轮决策

### 现象

active workflow 存在进程内字典中，key 是 `session_key`：

- `novel_agent/web/routes/chat.py:66` - `_ACTIVE_WORKFLOW_RUNS`
- `novel_agent/web/routes/chat.py:1160` - `_register_active_workflow`
- `novel_agent/web/routes/chat.py:1174` - `_ACTIVE_WORKFLOW_RUNS[session_key] = payload`

这意味着同一个会话再次注册 run 会覆盖前一个 run。没有 per-run 决策历史、没有修正事件队列，也没有“这条用户反馈应该应用到哪个合同/任务/文件对象”的关联模型。

### 影响

多轮聊天讨论后，系统很难稳定回答：

- 哪些讨论已经成为正式创作约束？
- 哪些讨论只是探索，不应落盘？
- 哪些反馈应该修改已有文件？
- 哪些反馈只影响后续任务？
- 当前任务池是否要重排？

### 为什么还没满足目标

目标里的“聊天讨论决定创作流程和方向”需要可追踪的决策状态，而不是只靠当前会话快照。

## P2-1：工作流状态持久化偏结果快照，不够表达进行中的交互式编辑

### 现象

聊天路径主要通过 workflow snapshot 返回新增/更新文件：

- `novel_agent/web/routes/chat.py:515` - merge `created_files`
- `novel_agent/web/routes/chat.py:520` - merge `updated_files`
- `novel_agent/web/routes/chat.py:1880` - `/chat/workflow-status`

前端在 done 后才稳定处理 auto-save 与文件聚焦：

- `novel_agent/web/static/app-core.js:2450` - 处理 `done`
- `novel_agent/web/static/app-core.js:2469` - `handleWorkflowAutoSave(evt.workflow)`

### 影响

文件变化更像“执行结束后的同步结果”，不是“聊天驱动的内容对象修改记录”。它没有明确表达：为什么改、根据哪轮讨论改、改了哪个创作对象、是否影响后续生成。

## 建议优先级

1. 先建立“聊天讨论 → 创作决策”的状态层，区分探索、确认、执行、修改、重排。
2. 建立统一可编辑对象模型，覆盖世界观、角色、事件线、大纲、细纲、章纲、正文、总结、合集等所有可创作内容。
3. 修流式 `collected_info` 写回，否则讨论阶段的信息沉淀不可靠。
4. 把 `provide_feedback` 接到合同、任务池、项目数据和具体文件的修改入口。
5. 为每次执行/修改引入 `run_id` 级别的决策与修正事件记录，避免同 session 覆盖。
6. 再考虑把执行 run 解耦为后台任务和真正的多 agent 并行认领。

## 验证建议

后续修复时至少补以下回归：

1. 流式讨论两轮后，`ChatSessionStore` 中的 `collected_info` 应包含新提取字段。
2. 用户说“先讨论，不要落盘”，系统不能创建或修改项目资料。
3. 用户说“按刚才讨论的方向开始”，系统应把讨论内容写入 `creation_contract` 和任务池。
4. 用户反馈“修改主角/题材/禁忌”后，`creation_contract`、`task_pool` 和相关项目资料应出现可验证变更。
5. 用户要求修改某一类已有内容时，系统应能定位并更新对应文件，例如角色、世界观、大纲、章纲或章节正文。
6. 修改已有内容后，后续生成任务应使用修改后的约束，而不是旧上下文。
7. 同一 session 连续触发多个决策/修改请求时，应绑定不同 `run_id` 或明确排队，不能静默覆盖。
