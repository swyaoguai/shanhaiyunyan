# 串行多 Agent 创作工作流设计

## 背景

当前创作助手已经具备 Router、Coordinator、Worldbuilder、Outliner、CharacterBuilder、ChapterWriter、Evaluator 等基础组件，但实际交互中仍容易出现两个问题：

1. 用户要求“创建世界观和角色卡”等复合任务时，系统可能只执行其中一部分。
2. 自然语言虽然被识别了，但没有稳定转化为可审查、可打回、可中断的创作工作流。

小说创作属于强上下文、强因果、强一致性的任务。世界观、人物、主线、大纲、章节正文之间存在严格依赖关系，因此不应默认采用并行多 Agent 写作。并行创作容易造成设定分叉、上下文不一致、角色动机冲突和章节因果断裂。

本设计目标是建立一个严谨的串行多 Agent 创作机制：由 Router 识别意图，由 Planner 拆解任务，由 Coordinator 串行调度子 Agent，由 Evaluator 独立审查产物，由用户随时中断并插入修改意见，最终把通过审查的内容写入资料库或正文。

## 核心原则

### 1. 串行优先

所有强依赖创作任务默认串行执行：

```text
用户需求
  -> 意图识别
  -> 工作流计划
  -> 上下文准备
  -> 专职 Agent 执行
  -> 独立审查
  -> 通过后入库或进入下一步
  -> 未通过则打回修订
```

禁止多个 Agent 同时生成互相依赖的世界观、角色、大纲或正文。

### 2. 单一事实源

所有 Agent 不直接依赖各自的隐式记忆，而是从统一的 `WorkflowContext` 读取上下文。`WorkflowContext` 是创作运行期间的唯一事实源，包含：

- 用户原始需求
- 已确认设定
- 当前资料库快照
- 上一步产物
- 审查意见
- 用户中断时给出的修改意见
- 冻结设定
- 禁止改动项
- 待解决冲突

每个 Agent 执行前都必须由 Coordinator 准备上下文包，执行后必须输出交接摘要。

### 3. 专职 Agent 只做一件事

子 Agent 只负责自己的阶段，不越权写其他资料：

- `RouterAgent`：识别用户意图，决定是否进入工作流。
- `WorkflowPlanner`：拆解任务链，生成可执行计划。
- `WorldbuilderAgent`：生成或修改世界观设定。
- `CharacterBuilderAgent`：生成或修改角色档案。
- `OutlinerAgent`：生成或修改大纲。
- `ProjectDataBuilderAgent`：生成或修改道具、事件线、细纲、章纲、自定义资料。
- `ChapterWriterAgent`：生成或修改指定章节正文。
- `EvaluatorAgent`：审查产物是否符合需求、上下文、资料库和前置设定。
- `Coordinator`：串行调度、状态推进、打回重试、保存产物。

### 4. 审查后提交

所有关键产物都必须先审查，再提交：

```text
Agent 输出草稿
  -> Evaluator 审查
  -> 通过：commit artifact
  -> 不通过：生成 revision request
  -> 打回原 Agent 修改
```

审查结果必须结构化保存，至少包括：

- 是否通过
- 问题等级
- 冲突点
- 缺失信息
- 需要打回的 Agent
- 修改要求
- 是否需要用户确认

### 5. 可中断与可恢复

用户可以在工作流任意阶段输入修改意见。系统必须：

1. 暂停当前工作流。
2. 把用户意见写入 `user_interruptions`。
3. 判断影响范围。
4. 标记受影响的任务和产物。
5. 由 Planner 重新规划后续任务。
6. 必要时打回前置 Agent 重做。
7. 恢复执行。

用户中断不是普通聊天，而是工作流事件。

class CreativeWorkflowRun:

## 标准执行流程

### 1. 自然语言入口

用户可以用自然语言触发任何资料库类别或正文任务：

- “创建世界观和角色卡”
- “生成道具物品：玄铁剑”
- “补一条势力阵营：合欢宗”
- “修改主角角色卡”
- “生成事件线”
- “写第 3 章正文”
- “把刚才的世界观改成末法时代”

Router 需要输出：

```json
{
  "intent": "creative_workflow",
  "target_categories": ["worldbuilding", "characters"],
  "operation": "create",
  "requires_workflow_plan": true
}
```

### 2. Planner 拆解任务

示例：用户说“先帮我写世界观和角色卡看看”。

Planner 应拆成：

```text
1. 准备创作上下文
2. Worldbuilder 生成世界观草稿
3. Evaluator 审查世界观
4. 通过后写入 worldbuilding.json
5. CharacterBuilder 基于世界观生成角色卡
6. Evaluator 审查角色卡
7. 通过后写入 characters.json
8. 汇总交接信息给用户
```

### 3. 执行与审查

每一步都执行：

```text
prepare_context
  -> execute_agent
  -> validate_output
  -> review_artifact
  -> commit_or_retry
  -> create_handoff
```

如果审查不通过：

```text
Evaluator 生成 revision request
  -> Coordinator 打回原 Agent
  -> Agent 根据审查意见修改
  -> Evaluator 再审
  -> 达到 max_retries 后请求用户确认
```

### 4. 用户中断

用户中途输入：“不对，主角不是复仇，是寻找失踪师父。”

系统应处理为：

```text
pause run
  -> record interruption
  -> update canonical context
  -> mark affected artifacts: character, outline, chapter_plan
  -> replan remaining tasks
  -> ask affected Agent revise
  -> resume
```

## 资料库类别入口

所有资料库类别都应拥有自然语言入口，包括用户自定义类别。

内置类别：

- `outline`：大纲
- `characters`：角色档案、角色卡、人设卡
- `worldbuilding`：世界观设定
- `items`：道具物品
- `eventlines`：事件线
- `detail_settings`：细纲设定
- `chapter_settings`：章纲设定
- `chapter_summary`：正文摘要
- `chapters`：正文章节

自定义类别：

- 从项目状态 `knowledge_categories` 动态读取。
- key 建议使用 `custom_*`。
- 自然语言匹配 name、key、id、aliases。
- 匹配后交给 `ProjectDataBuilderAgent` 或专用 Agent。

示例：

```text
用户：生成势力阵营 合欢宗，暗中控制边城商路
识别：custom_force / 势力阵营
执行：ProjectDataBuilderAgent
写入：custom_force.json
```

## 上下文传递规范

每个 Agent 接收的上下文必须包含：

- 当前任务目标
- 用户原始表达
- 相关资料库快照
- 已冻结设定
- 上一步 Agent 的 handoff
- 当前审查意见
- 输出格式要求

每个 Agent 输出必须包含：

- 正文产物或资料产物
- 新增事实
- 修改事实
- 依赖事实
- 风险点
- 给下一 Agent 的摘要

禁止只把完整聊天历史直接塞给 Agent。必须由 Coordinator 压缩成结构化上下文，否则上下文会越来越脏。

## 审查标准

Evaluator 至少检查：

1. 是否满足用户请求。
2. 是否写入了正确资料类别。
3. 是否与已确认设定冲突。
4. 是否引入未经确认的大设定。
5. 是否遗漏必要字段。
6. 是否对下一阶段足够可用。
7. 是否存在角色动机、世界规则、章节因果冲突。

审查输出示例：

```json
{
  "passed": false,
  "severity": "major",
  "issues": [
    {
      "type": "context_conflict",
      "message": "角色目标仍是复仇，但用户已改为寻找失踪师父。"
    }
  ],
  "revision_target": "CharacterBuilder",
  "revision_instructions": "重写主角目标和人物关系，不要再以复仇作为核心动机。",
  "requires_user_confirmation": false
}
```

## UI 与可观察性

前端应显示：

- 当前 run id
- 当前阶段
- 当前 Agent
- 任务队列
- 已完成任务
- 审查结果
- 打回原因
- 产物写入位置
- 暂停、继续、取消、插入修改意见

右侧助手消息不应只说“已完成”，而应能展示：

```text
工作流计划：
1. 世界观生成：已完成
2. 世界观审查：通过
3. 角色卡生成：执行中
4. 角色卡审查：等待中
```

## 不采用默认并行的原因

小说创作中的并行只适合低耦合任务，例如：

- 搜集参考素材
- 整理候选名字
- 生成多个备选标题
- 对同一产物进行多个独立评审

但以下任务不应并行：

- 世界观与角色卡同时生成
- 角色关系与主线大纲同时生成
- 多章节正文同时生成
- 章节正文与后续大纲修改同时生成

这些任务必须依赖统一上下文和前置产物，否则无法保证设定一致。

## 结论

本系统需要的不是“并行多 Agent”，而是“串行、可审查、可打回、可中断、可恢复、上下文统一”的多 Agent 创作工作流。

理想形态是：

```text
用户自然语言
  -> Router 识别目标资料/正文
  -> Planner 拆解任务链
  -> Coordinator 串行调度
  -> Specialist Agent 产出
  -> Evaluator 审查
  -> commit 或 retry
  -> Handoff 传递记忆
  -> 用户可随时中断修改
```

只有这样，才能保证长篇小说创作中的世界观、角色、事件线、大纲和正文始终处于同一个连续宇宙中。
