# 记忆系统改造方案

> 日期：2026-04-06  
> 项目：小说创作 Agent  
> 参考：Karpathy Wiki Memory System + agent 项目记忆系统演进方案（2026-02-09）

---

## 一、概述

### 现状问题

agent 项目现有记忆系统由四套组件构成：

| 组件 | 职责 | 状态 |
|------|------|------|
| SessionStore | 会话状态 + 版本管理（章节/角色/剧情点/version） | ✅ 稳定 |
| ContextManager | 上下文压缩 + 优先级策略 | ✅ 稳定 |
| AuxMemoryService | 结构化对象层（Resource/Item/Category/Relation）+ 回滚 + 检索 | ⚠️ 有基础，深度不足 |
| KnowledgeBase | 向量知识库（HybridSearch/UnifiedStore） | ⚠️ 有基础，Wiki 层缺失 |

**核心问题**：四套组件覆盖了"存储"和"检索"，但缺少一个**由 LLM 持续维护、可累积、可人工审阅的知识中间层**。现有系统的长期记忆更像"片段检索"，而不是"会成长的知识系统"。

具体表现：
- Ingest 分散在多个组件，没有统一的知识沉淀节点
- Query 依赖向量召回，没有"整理后认知优先"的策略
- Lint 偏局部（版本/回滚），不是针对记忆质量的系统性体检
- 各组件对同一事实可能有多份副本，缺乏统一的事实权威源

### 改造目标

在**不替换现有主链路**的前提下，增量引入 **WikiMemory 层**，让 LLM 持续维护一套 Markdown Wiki 作为知识中间层，实现知识复利增长。

---

## 二、设计原则

1. **不替换主链路**：SessionStore / ContextManager / KnowledgeBase 保持不变，作为底层存储和检索引擎
2. **增量演进**：WikiMemory 作为独立路径逐步接入，不破坏现有创作工作流
3. **Wiki 作为知识中间层**：Raw Source 必须经过 LLM 整理成 Wiki 页面，才能进入结构化层和向量层
4. **LLM 负责脏活**：总结、交叉引用、分类整理、保持一致性——人类做质量监督
5. **可回滚可审计**：Wiki 页和结构化数据都支持版本历史和回滚
6. **项目隔离**：WikiMemory 默认绑定 project_id，避免跨项目污染

## 三、三层架构设计

```
┌─────────────────────────────────────────────────────────┐
│  第一层：Raw Sources（原始资料层）                         │
│  不可变，LLM 只读不写                                      │
│  - 会话记录 / 文档 / 工具输出 / 用户输入                    │
└──────────────────────┬──────────────────────────────────┘
                       │ Ingest Phase 1
                       ▼
┌─────────────────────────────────────────────────────────┐
│  第二层：WikiMemory（知识中间层）[新增]                     │
│  LLM 持续编辑维护的核心层                                  │
│  - Markdown + frontmatter 页面                            │
│  - 由 LLM 生成、编辑、更新                               │
│  - 人类可读、可审阅、可干预                                │
└──────────────────────┬──────────────────────────────────┘
                       │ Ingest Phase 2
                       ▼
┌─────────────────────────────────────────────────────────┐
│  第三层：Schema + 结构化层                                │
│  规则文件 + AuxMemoryService + KnowledgeBase              │
│  - CLAUDE.md / AGENTS.md：Schema 规则约定                 │
│  - AuxMemoryService：Resource/Item/Category/Relation     │
│  - KnowledgeBase：向量索引 + 全文检索                     │
│  这些组件从 WikiMemory 同步数据，但不直接吃 Raw Source      │
└─────────────────────────────────────────────────────────┘
```

### 各层职责边界

| 层 | 谁能写 | 谁能读 | 核心价值 |
|---|--------|--------|---------|
| Raw Sources | 系统自动（会话/工具） | LLM + WikiMemory | 不可变的原始记录 |
| WikiMemory | LLM（主要）+ 人类（辅助） | 所有人 | 可累积的知识资产 |
| Schema/结构化层 | 受控流程（Schema 不允许 LLM 直接改） | Query 引擎 | 可检索的事实索引 |

### 层间交互原则

- **禁止**：Raw Source 直接落入结构化层（必须经 WikiMemory 整理）
- **允许**：WikiMemory 可直接同步到 AuxMemory / KnowledgeBase（单向）
- **禁止**：LLM 直接改 Schema 文件（CLAUDE.md / AGENTS.md）
- **允许**：Lint 发现问题时，生成修复建议由人类确认后执行

## 四、WikiMemory 层设计

### 页面格式：Markdown + frontmatter

每个 Wiki 页面是独立的 `.md` 文件，固定存放路径：`data/projects/<project_id>/wiki/<type>/<id>.md`

**frontmatter 块**（YAML）：
```yaml
---
id: project-vue-component-design        # 全局唯一标识
title: Vue 组件设计规范                  # 可读标题
type: Decision                          # 页面类型（见下方类型表）
project_id: proj_xxx                    # 所属项目
source_refs:                           # 来源引用（Raw Source ID 列表）
  - session_2026_04_01
  - doc_frontend_guide_v2
last_verified_at: 2026-04-06          # 上次验证时间
confidence: high                       # 置信度：high / medium / low
related_entities:                       # 关联实体（角色/模块/概念）
  - vue3
  - component-lib
  - frontend
open_questions:                         # 待确认问题
  - 是否需要支持异步组件？
change_log:                            # 变更历史
  - "2026-04-06: 初始创建（来源：session_2026_04_01）"
  - "2026-04-07: 补充组件拆分原则（来源：doc_frontend_guide）"
---
```

**正文固定块**（所有页面都必须有）：
```markdown
## Summary
（3-5 句话总结本页核心内容）

## Decisions
（已做出的决策及理由）

## Constraints
（约束条件、限制、边界）

## Known Facts
（已验证的事实，高置信度）

## Pending Uncertainties
（未验证的假设，待确认事项）
```

### 页面类型

| 类型 | 用途 | 典型生命周期 |
|------|------|-------------|
| `Project` | 项目级元信息、目标、范围 | 长期 |
| `Module` | 模块/功能的设计意图和约束 | 随项目演进 |
| `Decision` | 技术决策、架构选择及理由 | 凝固后不变 |
| `Constraint` | 剧情/角色/世界观约束（角色死亡、特殊设定） | 永久或剧情终结 |
| `UserPreference` | 用户偏好、风格倾向、创作习惯 | 持续更新 |
| `Workflow` | 创作流程、协作模式 | 随团队调整 |
| `Incident` | 踩坑记录、错误分析、修复记录 | 归档保留 |

### 生命周期

```
创建 → 验证 → 正常使用 → [定期 Lint] → 更新/合并/废弃
```

- **创建**：Ingest Phase 1 由 LLM 自动创建，或人类手动创建
- **验证**：新增页面需在 `last_verified_at` 后 7 天内由人类确认（或 LLM 标记 `confidence: high`）
- **正常使用**：被 Query 命中，作为上下文注入
- **Lint 触发更新**：孤立页面、过时信息、矛盾内容被 Lint 发现，触发 LLM 重写
- **废弃**：由 Lint 发现并经人类确认后标记为废弃（不删除，保留历史）

## 五、Ingest 两段式设计

### 第一段：Raw Source → WikiMemory

**触发时机**：
- 会话结束/阶段切换时（ContinuousWriter 章节完成时）
- 用户手动触发（"帮我整理一下这个项目的记忆"）
- 外部文档导入时
- 工具输出结果时（如 Search/Analysis 工具的输出）

**执行流程**：
```
Raw Source 输入
    ↓
LLM 判断：是否需要创建/更新 Wiki 页？
    ↓ 是
LLM 生成/更新 Wiki 页面
    - 提取关键信息（Decisions / Constraints / Facts）
    - 更新 frontmatter（source_refs / change_log / confidence）
    - 如有关联页面，同步更新 cross-ref
    ↓
Wiki 页面持久化
    ↓ 否（低价值片段）
直接进入结构化层（不占 Wiki 存储）
```

**职责划分**：LLM 是唯一写 Wiki 的主体，Raw Source 本身不可修改。

### 第二段：WikiMemory → 结构化层

**触发时机**：Wiki 页面创建/更新后自动触发，或定期批量同步。

**执行流程**：
```
WikiMemory 页面变化
    ↓
LLM 抽取结构化数据
    - 从 Wiki 抽取 Resource / Item 字段
    - 识别/更新 Category
    - 抽取 Relation（页面间关联）
    ↓
AuxMemoryService 同步
    - upsert Item / Resource / Relation
    - 不直接覆盖，由 LLM 确认后写入
    ↓
KnowledgeBase 向量同步
    - 页面正文入向量库（可选，取决于启用开关）
    - 不破坏现有章节向量数据
```

**关键约束**：WikiMemory 是结构化层的**数据源之一**，不是唯一来源。结构化层有自己的去重和合并策略，不因 Wiki 更新而自动覆盖已有数据。

## 六、Query 策略

### Wiki 优先的查询顺序

```
① 最近会话态（SessionStore 内存）
      ↓  未命中
② WikiMemory 层（优先找整理过的认知）
      ↓  未命中
③ AuxMemory 结构化关系（Resource / Item / Category）
      ↓  未命中
④ KnowledgeBase 向量召回（原始片段）
```

### 各类查询的优先级配置

| 查询类型 | 优先命中 | 说明 |
|---------|---------|------|
| 项目事实（角色设定、世界观） | WikiMemory → KB向量 | Wiki 已有整理 |
| 创作偏好（用户风格） | WikiMemory → AuxMemory | UserPreference 类型页 |
| 剧情约束（角色死亡、特殊事件） | WikiMemory | Constraint 类型页 |
| 章节内容（续写上下文） | SessionStore → KB向量 | 原始章节内容 |
| 工具结果（分析/搜索输出） | WikiMemory | 工具输出整理后沉淀 |
| 开放性问题（讨论/探索） | Raw Source 全文检索 | 尚未整理 |

### Query 响应中的 Wiki 回写

当 Query 的回答包含**高价值洞察**时，将该回答回存为新的 Wiki 页面：
```
Query 回答 → LLM 判断：是否值得回存 Wiki？
    ↓ 是
创建/更新 Wiki 页面
    - type: 推断或指定
    - source_refs: 包含本次 Query 的会话 ID
    - change_log: 记录"由 Query 回存"
```

## 七、Lint 流程

### 检查项

| 检查项 | 定义 | 处理方式 |
|--------|------|---------|
| **孤立页面** | 没有被任何页面引用（无 source_refs / related_entities） | 降权 或 标记待删除 |
| **重复页面** | 两页面 `Summary` 相似度 > 80% | 合并 或 标记冲突 |
| **过时页面** | `last_verified_at` 超过 30 天无更新 | 重验证 或 降 confidence |
| **无来源结论** | frontmatter 的 `source_refs` 为空 | 标记不可信 或 删除 |
| **矛盾内容** | 两页面对同一事实的描述冲突 | 保留两版本，标记冲突，由人类裁决 |
| **Relation 断链** | `related_entities` 引用了不存在的页面 | 补建 或 清除引用 |
| **高频未命中** | Query 多次未命中某主题，但无对应 Wiki 页 | 建议创建新页面 |
| **孤儿资源** | AuxMemory 的 Resource 有 item 引用但无 Wiki 来源 | 降级为原始片段 |

### Lint 输出形式

每次 Lint 生成一份报告，包含：
```json
{
  "timestamp": "2026-04-06T12:00:00Z",
  "issues": [
    {
      "type": "outdated_page",
      "page_id": "constraint-character-death-001",
      "severity": "high",
      "description": "页面超过 30 天未验证",
      "suggested_action": "re_verify",
      "action_detail": "联系人类确认该约束是否仍然有效"
    }
  ],
  "summary": {
    "total_pages": 142,
    "issues_found": 7,
    "high_priority": 2,
    "medium_priority": 3,
    "low_priority": 2
  }
}
```

### 修复动作

Lint 不直接执行修复，而是生成**可执行的修复建议**：
- `re_write`：LLM 自动重写页面内容
- `merge`：合并两个相似页面
- `delete`：标记删除（不物理删除，保留历史）
- `re_verify`：等待人类确认
- `create_cross_ref`：补充关联引用

人类可配置 Lint 的自动修复权限（完全自动 / 仅建议 / 禁用）。

## 八、落地计划

### Phase 1：WikiMemory 基础设施 + Ingest 产出

**目标**：让 Raw Source 能被 LLM 整理成 Wiki 页面。

**验收标准**：
- [ ] WikiMemory 页面 CRUD 可用（创建/读取/更新/删除）
- [ ] frontmatter 和正文格式符合规范
- [ ] Ingest Phase 1 接入 ContinuousWriter（章节完成后触发）
- [ ] 至少 3 个 Wiki 页面类型可用（Decision / Constraint / UserPreference）
- [ ] Wiki 页面持久化到 `data/projects/<project_id>/wiki/`

**不包含**：Query 优先读 Wiki / Lint / Phase 2/3 的功能

---

### Phase 2：Query 优先读 Wiki + Lint 流程

**目标**：Query 开始优先命中 Wiki 内容，系统具备自我体检能力。

**验收标准**：
- [ ] Query 策略实现 Wiki 优先命中
- [ ] Lint 流程可执行（支持全部检查项）
- [ ] Lint 报告可读可用
- [ ] Query 回答可回存 Wiki
- [ ] Ingest Phase 2 完成（Wiki → 结构化层同步）

**前置条件**：Phase 1 验收通过

---

### Phase 3：Schema/KB 反向校正 + 高级特性

**目标**：WikiMemory 与结构化层深度融合，Schema 可由 Lint 驱动更新。

**验收标准**：
- [ ] AuxMemory Item 可追溯到 Wiki 页面来源
- [ ] KnowledgeBase 与 WikiMemory 双向同步（可选）
- [ ] Schema（CLAUDE.md/AGENTS.md）可通过受控流程更新
- [ ] WikiMemory 支持版本历史和 diff
- [ ] 完整的项目隔离和多租户支持

**前置条件**：Phase 2 验收通过

## 九、与现有系统的集成

### 保持不变的组件

| 组件 | 原因 |
|------|------|
| SessionStore | 会话状态唯一权威源，WikiMemory 不覆盖 |
| ContextManager | 上下文压缩逻辑独立，不受 WikiMemory 影响 |
| KnowledgeBase（向量存储） | 现有章节内容存储不变，仅在 Phase 3 可选同步 Wiki |
| CLAUDE.md / AGENTS.md | Schema 规则文件不受 LLM 直接编辑 |

### 需要改造的接口

| 组件 | 改造点 |
|------|--------|
| ContinuousWriter | 章节完成后触发 Ingest Phase 1 |
| AuxMemoryService | Phase 2 新增"从 WikiMemory 同步"的入口 |
| NovelCoordinator | 嵌入 WikiMemory 查询调用（Phase 2） |
| RouterAgent | Query 时按策略优先查 WikiMemory |
| 整体 Web 层 | 新增 WikiMemory 可视化管理界面（参考 AuxMemory 模式） |

### 新增组件

| 组件 | 职责 |
|------|------|
| `WikiMemoryService` | Wiki 页面 CRUD、frontmatter 解析、持久化 |
| `WikiMemoryIngestor` | Ingest 两段式的编排逻辑 |

## 十、其他设计决策

### 为什么不替换 KnowledgeBase？

KnowledgeBase 是面向**创作内容**（章节、角色、世界观）的向量检索引擎，其数据和 WikiMemory 的用途不同。前者服务创作，后者服务元认知。两者共存且互补，不冲突。

### 为什么不直接让 LLM 改 Schema？

Schema 文件（CLAUDE.md / AGENTS.md）定义了系统级规则。一旦 LLM 可以直接编辑它们，就失去了稳定的规则锚点。WikiMemory 的设计让 LLM 在"知识层"发挥，同时保持"规则层"的确定性。

### 为什么 Wiki 页面用 Markdown 而不是数据库？

Markdown 对人类可读、可版本控制（Git）、可协作编辑（Obsidian / VS Code），这是 Karpathy 方案的核心洞察："知识应该是人类和 LLM 共同维护的"，而不是藏在数据库里。
