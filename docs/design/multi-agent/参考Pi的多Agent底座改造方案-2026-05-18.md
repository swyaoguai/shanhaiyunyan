# 参考 Pi 的多 Agent 底座改造方案

## Metadata

- Created: 2026-05-18
- Status: 设计草案
- Scope: 本地多 Agent 创作底座、生命周期事件、typed message/artifact、会话与执行轨迹
- Reference: `earendil-works/pi`，本次调研 commit `93b2e7f`
- Out of scope: 直接替换现有 Python/FastAPI 底座、A2A 协议、`a2a_adapter`、公网 agent 互通、把 Node 服务作为主运行时
- Primary goal: 保留现有本地多 Agent 架构，以 Pi 的运行时设计为参考，补齐事件协议、消息/产物协议、会话轨迹和可观测性。

## 1. 结论

`pi` 不适合直接套用为当前项目的主多 Agent 底座，也不建议把当前 Python/FastAPI 架构整体迁移到 `pi`。

推荐路线是：

```text
保留现有 Python 本地多 Agent 架构
  -> 借鉴 Pi 的 agent lifecycle event taxonomy
  -> 标准化 message / artifact / handoff / context_delta
  -> 将 TaskPool / Dispatcher / Router 的执行过程事件化
  -> 前端按 typed message/artifact 渲染
  -> 后续可选引入 JSONL event log / session tree / branch summary
```

关键判断：

| 维度 | Pi | 当前项目 | 结论 |
|------|----|----------|------|
| 技术栈 | TypeScript/Node agent harness | Python/FastAPI 本地应用 | 不适合主架构替换 |
| 默认目标 | coding-agent / single-agent harness | 小说创作多 Agent 工作流 | 业务语义不匹配 |
| 多 Agent | 不内置 sub-agents，需要 extension 自建 | 已有 Router/TaskPool/Dispatcher/MessageBus | 当前项目更贴合目标 |
| 事件流 | lifecycle events 较清晰 | progress/trace 已有但未统一 | 值得重点借鉴 |
| 会话 | JSONL tree、branch、compaction 设计成熟 | 项目状态和任务轨迹分散落盘 | 可借鉴为后续 event log |
| 前端 | typed message/artifact renderer 思路好 | 已有 workflow/task/handoff 展示雏形 | 可融合 |
| 成熟度 | AgentHarness 仍有 migration-ready 待办 | 项目主链已可运行 | 不宜整体依赖 Pi |
| 安全 | extensions/packages 有完整系统权限 | 当前本地应用边界更可控 | 不直接加载第三方 extension |

## 2. 当前底座基础

当前项目已经具备多 Agent 底座雏形，不是从零开始：

- `novel_agent/agents/router_agent.py`：入口意图判断、创作 workflow 触发、进度事件输出。
- `novel_agent/workflow/task_pool.py`：任务池、任务状态、依赖关系和优先级。
- `novel_agent/workflow/agent_dispatcher.py`：统一调度、路由、fallback、context snapshot、handoff、context delta、output validation。
- `novel_agent/agents/message_bus.py`：agent 间消息类型、发布订阅、请求响应、死信队列。
- `novel_agent/workflow/contracts.py`：任务定义、依赖、候选 agent、期望输出。
- `novel_agent/workflow/execution_context.py`：协作执行上下文和任务执行信封。
- `novel_agent/workflow/workflow_context.py`：workflow 产物、handoff、上下文承载。
- `novel_agent/web/static/app-core.js` 与 `continuous_write.js`：已经展示 workflow、task pool、execution trace、handoff 和子 agent 状态。

因此，主要短板不是“没有底座”，而是：

1. 事件协议不统一，progress、trace、workflow payload 多处各自组织。
2. message/artifact 类型未形成前后端共享的稳定 schema。
3. 任务执行生命周期没有像 `pi` 一样形成标准 event stream。
4. 会话轨迹和任务树可以回看，但还不是 append-only、可分支、可压缩的事件日志。
5. hooks / preflight / postprocess 机制未显式化，很多逻辑仍散落在 dispatcher 内部。

## 3. 从 Pi 借鉴什么

### 3.1 借鉴项

| Pi 设计 | 推荐落地方式 |
|---------|--------------|
| `agent_start` / `turn_start` / `message_*` / `tool_execution_*` / `agent_end` | 转成项目自己的 `AgentRuntimeEvent` 事件分类 |
| `toolExecution: parallel/sequential` | 在任务层表达 `execution_mode: serial/parallel`，创作强依赖默认 serial |
| `beforeToolCall` / `afterToolCall` | 在 Python 侧设计 `before_task_dispatch` / `after_task_result` hooks |
| Custom `AgentMessage` | 定义 typed message schema：text、workflow、task、artifact、handoff、context_delta、error |
| `pi-web-ui` message renderer registry | 前端建立 message/artifact renderer registry |
| JSONL session tree | 后续引入 append-only event log，支持任务分支与回放 |
| branch summary / compaction | 用于创作方案分支、章节试写分支、长上下文压缩 |
| RPC mode | 仅作为可选外部 coding-agent 工具，不进入主创作运行时 |

### 3.2 不借鉴或暂不引入

| Pi 能力 | 不建议原因 |
|---------|------------|
| 直接使用 `@earendil-works/pi-agent-core` 作为主底座 | 技术栈迁移成本高，业务语义不匹配 |
| 直接嵌入 `pi-coding-agent` 为创作 agent | 默认工具链偏代码任务，不适合小说创作主流程 |
| 加载 Pi extensions/packages | 权限边界过宽，第三方代码可执行任意系统操作 |
| 把 subagent extension 当核心多 Agent | Pi 本身不内置 sub-agent，示例能力不足以承载产品主链 |
| 引入 A2A 或外部互通层 | 当前项目定位是本地创作底座，明确不做外部 agent 互通 |

## 4. 目标架构

目标不是改成 `pi`，而是把当前底座升级为：

```text
Local Agent Runtime（Python）
  ├── RouterAgent
  ├── TaskPool
  ├── AgentDispatcher
  ├── AgentRuntimeEventBus
  ├── Message / Artifact / Handoff Schema
  ├── ContextDelta / OutputValidation
  ├── RuntimeEventLog
  └── Frontend Typed Renderer
```

推荐主链：

```text
User Input
  -> RouterAgent
  -> IntentAnalysis / CreationContract
  -> TaskDefinition
  -> TaskPool
  -> TaskExecutionEnvelope
  -> RoutingPolicy
  -> AgentDispatcher
  -> Agent.execute()
  -> OutputValidation
  -> Artifact / AgentHandoff / ContextDelta
  -> AgentRuntimeEvent
  -> RuntimeStateStore / EventLog
  -> Frontend Typed Renderer
```

核心原则：

1. 本地优先：所有主链仍在 Python 应用内完成。
2. 事件优先：重要状态变化必须有标准事件。
3. 类型优先：前后端围绕 message/artifact 类型协作，不靠散装 dict 猜字段。
4. 创作强一致：世界观、角色、大纲、章节等强依赖任务默认串行。
5. 可回放：事件日志能解释“谁在什么时候、基于什么上下文、产出了什么”。
6. 可降级：事件日志和 hooks 失败不应破坏主创作流程。

## 5. 核心协议设计

### 5.1 AgentRuntimeEvent

新增建议文件：

- `novel_agent/workflow/runtime_events.py`

建议结构：

```python
@dataclass
class AgentRuntimeEvent:
    event_id: str
    type: str
    timestamp: str
    run_id: str = ""
    task_id: str = ""
    parent_event_id: str = ""
    trace_id: str = ""
    agent_name: str = ""
    task_type: str = ""
    status: str = ""
    payload: Dict[str, Any] = field(default_factory=dict)
    metadata: Dict[str, Any] = field(default_factory=dict)
```

建议事件类型：

| 类型 | 说明 |
|------|------|
| `workflow_start` | 工作流开始 |
| `workflow_end` | 工作流结束 |
| `task_created` | 任务进入任务池 |
| `task_claimed` | 任务被某个 agent 认领或分配 |
| `task_start` | 任务开始执行 |
| `task_progress` | 任务执行中状态 |
| `task_end` | 任务完成 |
| `task_failed` | 任务失败 |
| `task_blocked` | 缺上下文或等待用户输入 |
| `route_start` | 路由决策开始 |
| `route_end` | 路由决策完成 |
| `message_start` | 对用户或内部流式消息开始 |
| `message_delta` | 流式内容增量 |
| `message_end` | 流式消息完成 |
| `artifact_created` | 产物生成 |
| `artifact_updated` | 产物更新 |
| `handoff_created` | 子 Agent 交接摘要生成 |
| `context_delta_created` | 上下文变化记录生成 |
| `validation_start` | 输出校验开始 |
| `validation_end` | 输出校验完成 |
| `fallback_start` | fallback 开始 |
| `fallback_end` | fallback 结束 |
| `user_input_required` | 需要用户确认或补充 |

### 5.2 Typed Message

新增建议文件：

- `novel_agent/web/schemas/messages.py`
- 或先在 `novel_agent/workflow/runtime_messages.py` 内实现轻量版本

建议消息类型：

```python
AgentRuntimeMessage = {
    "message_id": "uuid",
    "role": "user|assistant|system|agent|artifact|event",
    "type": "text|workflow|task|artifact|handoff|context_delta|error",
    "content": {},
    "created_at": "ISO8601",
    "trace_id": "...",
    "task_id": "...",
    "agent_name": "..."
}
```

前端不再只根据字符串或临时字段判断展示方式，而是：

```text
message.type
  -> renderer registry
  -> 对应 UI 组件
```

### 5.3 Typed Artifact

建议统一产物 envelope：

```python
ArtifactEnvelope = {
    "artifact_id": "uuid",
    "artifact_type": "worldbuilding|characters|outline|chapter|review|summary|context_bundle",
    "title": "",
    "content": {},
    "refs": [],
    "created_by": "WorldbuilderAgent",
    "task_id": "",
    "trace_id": "",
    "created_at": "ISO8601",
    "metadata": {}
}
```

现有 `Artifact`、`AgentHandoff`、`ContextDelta` 可以逐步适配这个 envelope，不需要一次性重写。

## 6. 分阶段实施计划

### P0. 文档与边界确认

目标：明确不替换底座、不做 A2A、不引入 Node 主运行时。

改动：

- 新增本方案文档。
- 后续如进入实现，在 `docs/todo/` 增加执行拆分清单。

验收：

- 方案明确区分“借鉴 Pi 设计”和“直接套用 Pi”。
- 文件级改造范围清晰。

### P1. 标准化 runtime event schema

目标：先统一事件名字和结构，不急着改前端。

建议改动：

- 新增 `novel_agent/workflow/runtime_events.py`。
- 定义 `AgentRuntimeEvent`、事件常量、`make_runtime_event()`。
- 在 `AgentDispatcher._append_execution_event()` 旁路生成标准事件。
- 保持现有 `collab_execution_trace` 兼容字段，避免前端断裂。

验收：

- `task_start/task_end/task_failed/route_end/fallback_*` 都能生成标准事件。
- 原有前端仍能读取旧字段。
- 新事件包含 `event_id/run_id/task_id/agent_name/type/timestamp/payload`。

推荐测试：

- `pytest novel_agent/tests -k dispatcher`
- 新增 runtime event 单元测试，覆盖序列化、缺省字段、兼容旧 trace。

### P2. Dispatcher 生命周期事件化

目标：把 `TaskPool/Dispatcher` 的真实执行过程变成稳定事件流。

建议改动：

- 在 `AgentDispatcher.dispatch()` 内补齐：
  - `route_start`
  - `route_end`
  - `task_claimed`
  - `task_start`
  - `validation_start`
  - `validation_end`
  - `artifact_created`
  - `handoff_created`
  - `context_delta_created`
  - `task_end`
  - `task_failed`
- 对 fallback 补齐 `fallback_start/fallback_end`。
- 将 `context_snapshot_id`、`required_context_keys`、`output_validation` 放进事件 payload。

验收：

- 任意一个章节任务可以从事件流还原完整执行链。
- 失败任务能看到失败前最后一个生命周期节点。
- fallback 任务能看到原 agent、fallback agent、fallback 原因。

推荐测试：

- `pytest novel_agent/tests/test_supervised_collab_foundation.py`
- 如测试文件不存在或名称变化，选择现有 dispatcher/task_pool 相关测试。

### P3. typed message/artifact 后端协议

目标：让后端输出给前端的 workflow/task/artifact/handoff/context_delta 有稳定类型。

建议改动：

- 新增或扩展消息 envelope。
- `RouterAgent._emit_progress()` 对 workflow 事件输出 type。
- `AgentDispatcher` 对任务产物输出 `ArtifactEnvelope`。
- `AgentHandoff`、`ContextDelta` 纳入统一 metadata 结构。

验收：

- 前端收到的协作事件至少有 `type`、`trace_id`、`task_id`、`agent_name`。
- task/handoff/context_delta/artifact 不需要通过猜字段决定 UI。
- 旧字段保留一轮，避免破坏已有页面。

推荐测试：

- 后端 schema 序列化测试。
- 现有 chat/continuous-write 路由测试。

### P4. 前端 typed renderer registry

目标：借鉴 `pi-web-ui` 的 renderer registry 思路，让前端展示从“条件堆叠”转为“类型分发”。

建议改动：

- 在 `app-core.js` 或独立模块新增：

```javascript
const messageRenderers = new Map();

function registerRuntimeMessageRenderer(type, renderer) {
  messageRenderers.set(type, renderer);
}

function renderRuntimeMessage(message) {
  const renderer = messageRenderers.get(message.type);
  return renderer ? renderer(message) : renderTextFallback(message);
}
```

- 首批 renderer：
  - `workflow`
  - `task`
  - `artifact`
  - `handoff`
  - `context_delta`
  - `error`
- `continuous_write.js` 的任务详情中补“信息链路”视图。

验收：

- 前端不再需要大量依赖 `payload.workflow || payload.task_pool || payload.collab_execution_trace` 的临时猜测。
- 用户能看到“输入上下文 -> 子 Agent -> 产物 -> 上下文变更 -> 下一步任务”。
- 旧消息仍有 fallback 展示。

推荐测试：

- `npm run test:frontend`
- 若只改单文件，可运行相关 vitest 文件。

### P5. append-only RuntimeEventLog

目标：为后续回放、分支、压缩做准备。

建议新增：

- `novel_agent/workflow/runtime_event_log.py`
- 每个项目目录下保存：

```text
data/projects/<project_id>/runtime_events/
  ├── current.jsonl
  └── archived/
```

事件日志原则：

1. append-only，不覆盖历史。
2. 每条事件一行 JSON。
3. 大文本只存摘要和 artifact ref。
4. 可从 JSONL 重建某次 workflow 的 task timeline。
5. 写日志失败不阻断主流程，但必须记录 warning。

验收：

- 能按 `trace_id` 读取一次完整 workflow。
- 能按 `task_id` 读取某任务完整生命周期。
- 能在项目重启后继续读取历史事件。

推荐测试：

- JSONL append/read/recover 测试。
- 半行损坏或非法 JSON 的恢复测试。

### P6. branch summary / compaction

目标：把 Pi 的 session tree 思路转成创作领域能力。

适用场景：

- 用户想比较两个世界观方向。
- 同一章生成多个版本。
- 长对话压缩为可追踪摘要。
- 某条任务分支废弃，但其有效设定需要保留。

建议设计：

```text
RuntimeEventLog
  -> branch_id
  -> parent_branch_id
  -> active_leaf_event_id
  -> branch_summary
  -> compaction_summary
```

第一版不需要完整 UI 树导航，只需要：

- 记录 `branch_id`。
- 支持“另开试写分支”。
- 支持“将当前分支总结为上下文包”。

验收：

- 同一章节可以保留多个生成分支。
- 用户选择其中一个分支提交时，其它分支不会污染主事实源。
- branch summary 可以转成 `ContextBundle` 或 `AgentHandoff`。

### P7. hooks / policy extension

目标：把 dispatcher 内部的 preflight/postprocess 规则外置成可组合 hook，但保持本地可信代码边界。

建议 hook：

```python
before_route(event, context) -> Optional[RouteOverride]
after_route(event, context) -> None
before_task_dispatch(event, context) -> Optional[TaskBlock]
after_task_result(event, context) -> Optional[ResultPatch]
before_artifact_commit(event, context) -> Optional[ReviewRequired]
after_context_delta(event, context) -> None
```

限制：

- 第一版只允许项目内 Python hook。
- 不加载第三方脚本。
- 不做热加载 extension。
- hook 异常默认转为事件，不直接吞掉主流程错误。

验收：

- output validation、人工确认、敏感覆盖检查可以作为 hook 注册。
- hook 执行顺序可预测。
- hook 结果进入 runtime event。

必测：

- 任务详情“信息链路”展示。

推荐命令：

```bash
npm run test:frontend
```

### 手动验收

1. 创建一个长篇项目。
2. 触发世界观、角色、大纲、章节任一多 Agent 流程。
3. 前端可看到当前 agent、任务状态和产物。
4. 后端项目目录可看到 runtime event 记录。

## 9. 风险与缓解

| 风险 | 说明 | 缓解 |
|------|------|------|
| 事件层过度设计 | 太早追求完整 tree/session 可能拖慢主线 | P1-P4 先做稳定事件和 typed renderer |
| 前端兼容破坏 | 旧 payload 仍在使用 | 至少保留一轮旧字段和 fallback renderer |
| 日志膨胀 | 大文本进入 JSONL 会迅速变大 | 大文本只存摘要、hash、artifact ref |
| hook 滥用 | hook 过早开放会变成隐式控制流 | 第一版只做项目内固定 hook |
| 创作并行导致事实分叉 | 多 Agent 并行写强依赖内容会冲突 | 强依赖任务默认 serial，parallel 只用于检索/审查类 |
| 引入 Node 复杂度 | 直接引 Pi 会多运行时、多状态源 | Node 只作为可选外部 coding-agent，不进主链 |

## 10. 明确不做

本轮和下一轮底座改造不做：

1. 不实现 A2A。
2. 不新增 `a2a_adapter`。
3. 不做公网 agent 互通。
4. 不把 `pi` 作为主运行时。
5. 不把 `pi` extensions/packages 直接加载进产品。
6. 不把小说创作强依赖任务默认并行化。
7. 不重写 Router/Coordinator/Dispatcher 主链。

## 11. 推荐开发顺序

最小可行路径：

```text
P1 runtime_events.py
  -> P2 AgentDispatcher 生命周期事件
  -> P3 typed message/artifact envelope
  -> P4 前端 renderer registry
  -> P5 RuntimeEventLog
```

每一步都应保持可独立发布：

1. 先让事件标准化，但不改变 UI。
2. 再让 UI 读取 typed message，但保留旧 payload。
3. 再把事件持久化为 JSONL。
4. 最后做 branch summary / compaction。

## 12. 最终目标验收

当这一轮改造完成后，应达到：

- 用户能看懂每个子 Agent 为什么执行。
- 开发者能从事件日志还原一次完整多 Agent 运行。
- 每个任务都有路由原因、输入上下文、输出校验、产物引用和上下文变化。
- 前端能按类型展示 workflow/task/artifact/handoff/context_delta。
- 未来新增 agent 时，只需声明能力、任务类型、输入输出和 renderer，不需要到处写临时字段判断。

这条路线的核心价值是：不把已有能跑的底座推倒，而是把它变成一个更稳、更可解释、更可扩展的本地 agent runtime。
