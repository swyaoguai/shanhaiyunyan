# 监督式自组织多 Agent 演进方案

## 1. 文档目标

本文档用于指导当前项目从“强编排型多 Agent”逐步演进为“监督式自组织多 Agent”。

适用范围：

- 当前项目的协作式小说创作链路
- 用户讨论 → 方案确认 → 自动分配 Agent → 自动创作 的目标流程
- 后续的架构升级、协议设计、数据结构设计与开发排期

本文档不是概念讨论，而是后续开发时的落地参考。

---

## 2. 当前系统现状

### 2.1 当前多 Agent 的真实运行方式

当前项目的多 Agent 不是基于独立进程集群，也不是基于外部自治 Agent 框架，而是运行在单个 Python Web 应用进程内部。

当前核心运行基础：

- Web 启动入口：[`main()`](../run.py:136)
- Agent 基类：[`BaseAgent`](../novel_agent/agents/base_agent.py:37)
- 路由层：[`RouterAgent`](../novel_agent/agents/router_agent.py:63)
- 编排层：[`NovelCoordinator`](../novel_agent/workflow/coordinator.py:118)
- 消息通信层：[`MessageBus`](../novel_agent/agents/message_bus.py:101)
- 聊天入口层：[`chat()`](../novel_agent/web/routes/chat.py:1377)
- 章节协作主链：[`_write_single_chapter_internal()`](../novel_agent/workflow/coordinator.py:1540)

### 2.2 当前架构的本质

当前系统本质上是：

**一个基于 `Python asyncio + FastAPI + Coordinator-Worker` 的应用内多 Agent 系统。**

其特点：

- `Router` 负责识别意图和选择执行链
- `Coordinator` 负责统一编排流程
- 各子 Agent 负责单点任务执行
- 状态、文件、上下文由统一管理器维护
- 消息总线已经存在，但目前不是主链的核心驱动方式

### 2.3 当前架构的优点

| 优点 | 说明 |
| :--- | :--- |
| 可控 | 所有主流程在编排器里，出问题容易定位 |
| 可测 | 章节链和工作流可以做集成测试 |
| 可落盘 | 文件、状态、摘要都能稳定持久化 |
| 可产品化 | 当前架构已经适合前端展示执行阶段和产物 |
| 可扩展 | 新子 Agent 可以继续接入现有主链 |

### 2.4 当前架构的代价

| 代价 | 说明 |
| :--- | :--- |
| Coordinator 过重 | 太多决策、流程与异常处理都堆在一个编排器里 |
| Agent 自治性不足 | 大部分 Agent 只是被动被调用，不能主动协作 |
| MessageBus 利用不足 | 已有总线，但主链并未真正依赖协议协作 |
| 协作协议薄弱 | 当前更多是函数调用，不是 Agent 间的明确任务协议 |
| 未来复杂度会抬升 | 随着功能增多，Coordinator 会越来越难维护 |

## 3. 目标架构定义

## 3.1 不建议的目标

不建议将项目直接改造成“完全无中心自治多 Agent”。

原因：

- 小说创作是强产品化场景，不是纯研究型自治实验
- 用户需要：
  - 可解释
  - 可暂停
  - 可恢复
  - 可查看阶段状态
  - 可强制中断
  - 可人工确认关键节点
- 完全自治会带来：
  - 执行不可预测
  - 状态漂移
  - 调试困难
  - 成本失控
  - 结果不稳定

## 3.2 推荐目标

推荐目标是：

# 监督式自组织多 Agent

定义如下：

- 用户先通过对话确认创作方案
- 系统把讨论结果沉淀为正式任务合同
- Agent 根据能力和依赖关系自主认领任务
- Agent 之间通过协议和消息协作
- Coordinator 不再手写每一步执行链，而是：
  - 管理任务池
  - 监督任务状态
  - 控制预算与超时
  - 仲裁冲突
  - 在必要时请求用户确认
  - 负责最终收敛与落盘

### 目标状态的核心特征

| 特征 | 当前 | 目标 |
| :--- | :--- | :--- |
| 任务分配 | Coordinator 直接点名调用 | Agent 按能力自主认领 |
| 执行方式 | 固定顺序链 | 事件驱动任务图 |
| Agent 协作 | 少量、弱协议 | 明确消息协议 |
| 状态管理 | 编排器集中维护 | 编排器维护总状态，Agent 自主推进局部任务 |
| 用户参与 | 入口和中断控制 | 入口、方案确认、关键节点确认 |
| Coordinator 角色 | 全知全控执行者 | 监督者、仲裁者、收敛控制器 |

## 4. 总体演进原则

## 4.1 不推翻重做

当前基础设施已经具备演进条件：

- [`BaseAgent`](../novel_agent/agents/base_agent.py:37)
- [`MessageBus`](../novel_agent/agents/message_bus.py:101)
- [`RouterAgent`](../novel_agent/agents/router_agent.py:63)
- [`NovelCoordinator`](../novel_agent/workflow/coordinator.py:118)

因此不建议重构成全新框架，而应当：

**保留现有主链可用性，在其上渐进升级。**

## 4.2 先协议化，再自治化

错误路径：

- 先做“完全自治”
- 再补协议
- 再补测试
- 再补状态

正确路径：

1. 先定义任务合同
2. 再定义协作消息协议
3. 再定义 Agent 能力模型
4. 再让部分 Agent 自主认领
5. 最后扩大到全链自治协作

## 4.3 始终保留人工干预点

以下节点必须保留人工干预能力：

- 方案确认
- 重大重规划
- 预算异常
- 风格冲突
- 结果明显跑偏
- 关键产物提交前审阅

## 5. 核心新增设计

# 5.1 创作任务合同层

当前用户讨论结果更多以 `collected_info` 和上下文形式存在。

后续必须新增一个正式对象，例如：

- `CreationContract`
- 或 `CollabTaskPlan`

建议结构：

```json
{
  "contract_id": "uuid",
  "goal": "创作一部长篇小说",
  "user_confirmed": true,
  "scope": {
    "novel_type": "玄幻",
    "theme": "复仇成长",
    "target_length": "长篇"
  },
  "constraints": {
    "style": ["压抑", "递进", "不低俗"],
    "forbidden": [],
    "quality_rules": ["避免AI腔", "保证连贯性"]
  },
  "deliverables": [
    "worldbuilding.json",
    "outline.json",
    "chapters/*.md",
    "stage_summaries/*.md"
  ],
  "agent_candidates": [
    "Worldbuilder",
    "Outliner",
    "CharacterBuilder",
    "ContextStrategy",
    "ContentReader",
    "ChapterWriter",
    "Evaluator",
    "Polisher",
    "ContentExpansion",
    "FileNaming",
    "SummaryOrchestrator"
  ],
  "task_graph": [],
  "created_at": "ISO8601",
  "updated_at": "ISO8601"
}
```

### 作用

- 作为所有 Agent 的统一任务依据
- 作为后续任务市场的输入
- 作为用户确认后的正式执行契约
- 作为恢复、重规划、审计的依据

### 建议落盘位置

可以保存到：

- `client_state/creation_contract.json`
- 或独立文件 `creation_contract.json`

优先建议使用项目状态接口保存：

- [`save_project_state()`](../novel_agent/project_manager.py:282)

---

# 5.2 Agent 能力声明层

当前 Agent 已经是独立类，但还没有“能力声明”。

后续建议每个 Agent 增加标准化能力元信息，例如：

```python
{
    "agent_name": "ContextStrategy",
    "capabilities": ["plan_context", "identify_dependencies"],
    "accept_task_types": ["context_plan"],
    "required_inputs": ["creation_contract", "chapter_number"],
    "produced_outputs": ["read_plan"],
    "priority": 80,
    "max_concurrency": 2
}
```

建议在 [`BaseAgent`](../novel_agent/agents/base_agent.py:37) 上增加统一接口：

- `get_capabilities()`
- `accepts_task(task) -> bool`
- `estimate_cost(task) -> dict`
- `requires_inputs(task) -> list`
- `produces_outputs(task) -> list`

### 作用

- 让系统根据能力自动匹配 Agent
- 让 Agent 可以“自主认领任务”
- 为任务市场提供基础条件
- 为后续可视化能力图谱提供数据源

---

# 5.3 任务市场层

新增一个概念层：

## Task Market / Task Pool

作用：

- Coordinator 发布任务到任务池
- Agent 根据能力声明判断是否认领
- 任务池管理状态流转

任务状态建议：

- `pending`
- `claimed`
- `running`
- `blocked`
- `review_required`
- `completed`
- `failed`
- `aborted`

建议任务结构：

```json
{
  "task_id": "uuid",
  "task_type": "write_chapter",
  "status": "pending",
  "priority": 90,
  "depends_on": ["context_ready_task_id"],
  "inputs": {
    "chapter_number": 1,
    "chapter_title": "旧城归来"
  },
  "assigned_agent": "",
  "candidate_agents": ["ChapterWriter"],
  "result_ref": "",
  "retry_count": 0,
  "created_at": "ISO8601",
  "updated_at": "ISO8601"
}
```

### 建议落盘

- `client_state/task_pool.json`
- 或 `workflow_tasks.json`

---

# 5.4 消息协议层

当前 [`MessageBus`](../novel_agent/agents/message_bus.py:101) 已支持基础消息收发，但要支撑自组织协作，需要新增更贴近业务的消息类型。

建议在 [`MessageType`](../novel_agent/agents/message_bus.py:24) 基础上扩展：

- `TASK_PROPOSED`
- `TASK_CLAIMED`
- `TASK_REJECTED`
- `TASK_BLOCKED`
- `DEPENDENCY_REQUIRED`
- `DEPENDENCY_RESOLVED`
- `REVIEW_REQUESTED`
- `REVIEW_ACCEPTED`
- `REPLAN_REQUIRED`
- `PLAN_APPROVED`
- `PLAN_ABORTED`

### 示例协作过程

1. 主交互 Agent 生成合同
2. Coordinator 广播 `TASK_PROPOSED(context_plan)`
3. ContextStrategy 发送 `TASK_CLAIMED`
4. ContextStrategy 完成后发布 `DEPENDENCY_RESOLVED(context_ready)`
5. ChapterWriter 检测到条件满足后认领 `write_chapter`
6. Evaluator 自动认领评估任务
7. 若评估失败，Polisher 认领修订任务
8. 若全部通过，FileNaming / SummaryOrchestrator 认领收尾任务

### 作用

将当前函数串行调用，转为：

**可观察、可审计、可调试、可扩展的协作协议流。**

## 6. Coordinator 的角色重构

## 6.1 当前角色

当前 [`NovelCoordinator`](../novel_agent/workflow/coordinator.py:118) 同时扮演：

- 任务拆解者
- 调度者
- 执行顺序定义者
- 状态管理者
- 文件写入者
- 异常控制器

这会导致过重。

## 6.2 目标角色

后续重构后，Coordinator 建议只负责：

- 任务合同装载
- 任务池维护
- 任务状态机推进
- 预算与并发控制
- 超时、失败、冲突处理
- 与用户确认节点交互
- 产物收敛
- 最终落盘与项目状态总结

### 不再直接负责的事情

不再由 Coordinator 直接手写：

- 哪个 Agent 先跑
- 哪个 Agent 后跑
- 哪个 Agent 是否参与
- 子 Agent 的具体调用顺序

这些改成：
- 任务图
- 依赖关系
- Agent 自主认领

---

## 7. 用户交互流程重构

你提出的目标是：

> 主要就是通过跟用户讨论交互，确定方案后进行自动分配 agent 进行创作

这要求前台流程重构成三段式：

# 7.1 第一阶段：讨论阶段

由主交互 Agent 负责：

- 提问
- 补全缺失需求
- 归纳偏好
- 提炼用户约束
- 组织成方案草案

输出：
- 草案版 `CreationContract`

# 7.2 第二阶段：方案确认阶段

系统向用户展示：

- 创作目标
- 风格约束
- 输出内容
- 计划启用的 Agent
- 预计任务图

用户确认后：

- `user_confirmed = true`
- 正式进入执行阶段

# 7.3 第三阶段：自动协作执行阶段

执行阶段：

- Coordinator 初始化任务池
- 各 Agent 根据能力自主认领任务
- 关键节点向用户回显
- 产物逐步落盘

### 必须保留的用户确认节点

建议保留：

- 合同确认
- 大纲确认
- 重大重规划确认
- 终稿汇总确认

## 8. 推荐的目标执行流

以下是建议的目标工作流。

## 8.1 顶层流

```text
用户发起创作
  -> 主交互 Agent 讨论需求
  -> 形成方案草案
  -> 用户确认方案
  -> Coordinator 创建任务池
  -> Agent 自主认领任务
  -> 产物逐步写入项目
  -> 关键节点回显
  -> 最终收敛与完成
```

## 8.2 章节级流

```text
合同已确认
  -> 发布 context_plan 任务
  -> ContextStrategy 认领
  -> 发布 content_read 任务
  -> ContentReader 认领
  -> 发布 write_chapter 任务
  -> ChapterWriter 认领
  -> 发布 evaluate_chapter 任务
  -> Evaluator 认领
      -> 若通过:
         -> 发布 file_naming 任务
         -> 发布 summary_orchestrate 任务
      -> 若不通过:
         -> 发布 polish_chapter 任务
         -> 或发布 expand_content 任务
         -> 再次评估
```

## 10. 推荐的开发顺序

不建议一次性全部改完，建议按阶段实施。

# 阶段一：协议化现有强编排

目标：
- 不改变用户体验
- 不改变主链产物
- 先把现有调用链“结构化”

开发项：

1. 新增 `CreationContract`
2. 新增 `TaskDefinition`
3. 在现有创作链中生成合同和任务图草案
4. 将当前章节内部各步骤都包装成任务对象

### 阶段一完成标准

- 当前功能不退化
- 可以把现有章节链导出为任务图
- 后续自治化具备基础

# 阶段二：能力声明化

开发项：

1. 为各 Agent 增加能力声明
2. 新建能力注册表
3. 让系统可查询“某任务由谁能接”

### 阶段二完成标准

- 任意任务都能通过能力表找到候选 Agent
- 不再依赖硬编码“只能这个 Agent 做”

# 阶段三：部分自治试点

优先试点章节内部链：

- `ContextStrategy`
- `ContentReader`
- `Evaluator`
- `Polisher`
- `SummaryOrchestrator`

不建议先从：
- `Worldbuilder`
- `Outliner`
- `Router`

开始试点，因为风险更高。

### 阶段三完成标准

- 章节内部至少一段链路不再由 Coordinator 直接点名执行
- 任务认领与结果回传可运行

# 阶段四：讨论 → 合同 → 自治执行闭环

开发项：

1. 主交互 Agent 输出正式合同草案
2. 增加用户确认接口
3. 用户确认后自动创建任务池
4. Agent 自主认领执行
5. Coordinator 只做监督和收敛

### 阶段四完成标准

- 用户可以通过讨论确定方案
- 系统自动进入执行
- 协作过程可回显、可暂停、可恢复

## 12. 风险与控制手段

## 12.1 主要风险

| 风险 | 说明 |
| :--- | :--- |
| Agent 相互循环依赖 | 反复请求上下文或重规划 |
| 成本失控 | 任务协商轮数过多 |
| 状态漂移 | 多 Agent 视角不一致 |
| 跑偏 | 自组织后任务边界不清 |
| 难调试 | 问题不再是一条线性调用链 |

## 12.2 控制手段

必须增加：

- 最大协商轮次
- 最大任务重试次数
- Agent 接单超时
- 关键节点用户确认
- 冲突强制仲裁
- 失败回退到强编排模式

### 推荐保底策略

新增：

- `supervised_mode = true`
- `fallback_to_orchestrated = true`

一旦自组织流程异常，就退回当前稳定的强编排执行链。

---

## 13. 最终推荐结论

**建议将项目演进为“监督式自组织多 Agent”，而不是完全去中心化自治。**

推荐原因：

1. 更适合创作产品
2. 更适合用户可见流程
3. 更适合断点续作与状态展示
4. 更适合后续测试与排障
5. 与当前项目基础设施兼容度高

用一句话总结：

> 保留 [`NovelCoordinator`](../novel_agent/workflow/coordinator.py:118) 作为监督与收敛控制层，把当前对子 Agent 的硬编码顺序调用，逐步替换成基于 [`MessageBus`](../novel_agent/agents/message_bus.py:101) 的任务合同、能力声明、任务认领、依赖协商与结果提交机制。
