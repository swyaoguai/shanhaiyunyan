# 小说转剧本转换器模块开发方案

## 相关文件
- `novel_agent/web/templates/index.html`：极左资源栏模块入口定义，新增模块按钮要在这里注册。
- `novel_agent/web/static/app-core.js`：模块切换总入口，决定左侧模块如何渲染主工作区。
- `novel_agent/web/static/app-nav.js`：左侧导航栏渲染入口，决定各模块的导航标题、操作按钮和导航内容。
- `novel_agent/web/static/app-chapters.js`：协作创作模式的“导入小说”弹窗与前端调用模式，可复用其导入交互。
- `novel_agent/web/static/continuous_write.js`：无限续写模块的独立导航、导入会话、导出能力，可复用其模块化拆分方式。
- `novel_agent/web/static/short-story/short-story-render.js`：固定面板式模块的渲染模式，适合作为“转换器”新模块的前端模板。
- `novel_agent/web/routes/projects.py`：协作模式的导入路由，已具备 `txt/md/docx` 文件接入和项目级保存逻辑。
- `novel_agent/web/routes/continuous_write.py`：独立模式导入路由，已具备按会话隔离的导入模式。
- `novel_agent/novel_import_service.py`：小说文件解析与章节拆分公共服务，当前最值得复用。
- `novel_agent/tests/test_mode_import_memory.py`：现有导入链路测试样例，可作为新模块测试模板。

## 1. 目标

在左侧新增一个独立模块，支持用户：

1. 直接粘贴小说正文。
2. 导入已写好的小说文件。
3. 将小说内容转换为“剧本格式”。
4. 在页面内查看、编辑、复制、导出转换结果。

这个模块更适合做成“独立转换工作台”，而不是挂到现有协作创作或无限续写里。原因是它的目标不是继续写小说，而是把已有文本结构化重写为另一种交付格式。

## 2. 结合现有架构的建议

### 2.1 模块形态建议

建议新增独立模块 ID：`novel-to-script`

放置位置建议：
- 位于左侧资源栏，放在 `短篇创作` 与 `协作创作` 之间，语义上更自然。

前端实现方式建议：
- 采用“像短篇创作一样的独立工作台”。
- 不建议做成“像协作创作那样的章节编辑器”。

原因：
- 当前 `short-story` 模块已经是一个自包含面板式流程模块，适合承载“输入 -> 转换参数 -> 执行 -> 结果预览 -> 导出”。
- 当前 `write` 模块的核心是章节协作编辑，不适合承载一次性格式转换任务。
- 当前 `continuous_write` 虽然也有独立状态和导入能力，但其数据模型偏“续写会话”，与“转换任务”不完全一致。

### 2.2 后端实现方式建议

建议新增独立路由文件：
- `novel_agent/web/routes/novel_to_script.py`

建议新增独立服务文件：
- `novel_agent/novel_to_script_service.py`

公共能力复用：
- 文件导入和章节拆分：复用 `NovelImportService.parse_novel_file()`
- 项目级状态保存：复用现有 `project-state/batch-get` / `project-state/batch-set` 或新增专用接口
- 导出文本 / docx：优先参考 `continuous_write.py` 与 `short_story.py` 的导出实现

## 3. MVP 功能范围

第一版建议只做“从小说到分场景剧本”的单向转换，不做影视工业级拆解，不做镜头脚本，不做分镜。

### 3.1 输入能力

支持两种输入方式：

1. 手动输入 / 粘贴
   - 文本框直接粘贴小说内容
   - 适合短篇、片段、单章试转

2. 文件导入
   - 支持 `.txt` / `.md` / `.docx`
   - 直接复用现有导入限制和解码逻辑

建议第一版不要支持：
- `.pdf`
- `.epub`
- 多文件批量导入

原因：
- 当前公共导入服务只稳定支持 `txt/md/docx`
- PDF/EPUB 解析噪声高，会明显拉高异常率和维护成本

### 3.2 转换目标格式

你补充的文字示例和截图已经把目标格式限定得很明确了。第一版不应该输出“标准影视工业剧本”，而应该输出更接近“网文改编场景台本”的格式。

这个格式的特点是：
- 以场景块为核心，而不是以章节摘要或人物表为核心。
- 每个场景都重复声明 `人物`、`环境`、`动作/旁白`，便于直接阅读和继续改写。
- 角色台词行允许带括号说明，例如 `夜天（内心独白，语气绝望，低声呢喃）：...`
- 非人物发言也允许独立成行，例如 `闪回片段（快速切换，语气凶狠）：...`、`动作/音效：...`
- 整体上是“小说叙述 + 台词标签 + 表演提示”混合体，不是纯对白剧本。

建议第一版默认输出这种格式：

```text
【场景一：青云门后山茅厕 - 暴雨夜】
人物：夜天（青云门杂役弟子，二十岁左右，衣衫破旧，面带疲惫）
环境：简陋茅厕，木门破旧，被风吹得“吱呀”作响，地面积满污水，混杂着雨水、泥土与异味，窗外暴雨倾盆，雷声隐隐
动作/旁白：夜天蹲在茅厕角落，手里攥着一个破瓢，佝偻着身子，费力地往外舀着地上的积水，眉头紧锁，满脸嫌弃与无奈。
夜天（内心独白，语气绝望，低声呢喃）：我真的只想躺平啊……
夜天（动作）：舀水的动作顿了顿，肚子传来“咕咕”的叫声。
闪回片段（快速切换，语气凶狠）：刘管事指着夜天的鼻子，唾沫星子飞溅：“再敢偷懒耍滑，就把你剁了丢去后山喂灵兽！”
动作/音效：突然，一道黑色闪电划破天际，“咔嚓”一声巨响。

【场景二：青云门后山茅厕 - 暴雨渐歇】
人物：夜天、小黑猫（巴掌大小，通体漆黑，毛发油亮，眼神高冷，自带傲慢气场）
环境：同场景一，积水依旧，异味未散，暴雨渐小，天边透出一丝微光
动作/旁白：不知过了多久，夜天的手指微微动了动，意识逐渐清醒。
小黑猫（灵魂传音，语气苍老、傲慢，带着不耐烦，直接传入夜天脑海）：凡人，能成为本喵祖的肉垫，是你八辈子修来的福分。
```

第一版输出至少包含：
- 场景标题，格式固定为 `【场景X：地点 - 时间/天气】`
- 人物
- 环境
- 动作/旁白
- 带角色名的动作或对白行
- 必要时的 `动作/音效`
- 必要时的 `闪回片段`

第一版可不强制包含：
- 镜头号
- 景别
- 调度说明
- 拍摄技术提示
- 单独的全局人物表

### 3.3 示例与截图分析结论

根据你给的文字版示例和截图，这个模块的“最终导出文本”需要尽量贴近以下规范：

1. 场景标题使用全角书名式括号
   - 格式为 `【场景一：地点 - 时间】`
   - 编号建议用中文数字 `一、二、三`

2. 场景内部字段顺序尽量固定
   - `人物`
   - `环境`
   - `动作/旁白`
   - 角色动作/对白/内心独白
   - `闪回片段`
   - `动作/音效`

3. 文本是“逐行块状排版”，不是表格，也不是卡片拼接
   - 截图里每一行都是一个独立的表意单元
   - 所以前端结果区应优先支持纯文本连续预览，而不是强依赖复杂卡片 UI

4. 角色行的标签要保留表演信息
   - 如 `夜天（内心独白，语气绝望，低声呢喃）：...`
   - 如 `夜天（动作）：...`
   - 如 `小黑猫（灵魂传音，语气不屑）：...`

5. 非角色叙事标签要允许混排
   - `动作/旁白`
   - `动作/音效`
   - `闪回片段`

6. 输出重点不是“压缩”，而是“可演”
   - 要把小说原文中的心理、动作、环境拆成适合台本阅读的行
   - 允许保留网文化语气，只要阅读顺畅、场景清楚

7. 第一版默认风格建议直接命名为
   - `scene_block_webnovel_script`
   - 中文展示名可写成：`网文场景台本`

## 4. 推荐的用户流程

### 4.1 页面流程

建议页面分成四块：

1. 输入区
   - 粘贴小说
   - 导入文件
   - 显示字数与章节识别结果

2. 转换配置区
   - 剧本类型：通用场景剧本 / 对话强化版 / 网文短剧版
   - 输出粒度：按全文转换 / 按章节转换
   - 场景密度：低 / 中 / 高
   - 对白占比：低 / 中 / 高
   - 人名保留策略：保留原名 / 模糊修正

3. 转换结果区
   - 左侧显示章节/场景目录
   - 右侧显示剧本正文
   - 支持局部编辑

4. 操作区
   - 开始转换
   - 重新转换
   - 复制结果
   - 导出 TXT / MD / DOCX

### 4.2 交互建议

导航栏建议不要做成复杂目录树，第一版只需要：
- `转换面板`
- `转换结果`
- `历史草稿`（可选，MVP 可以先不做）

如果按短篇模块方式实现，左侧导航甚至可以只保留两个入口：
- `工作台`
- `成稿`

## 6. 后端拆分建议

### 6.1 服务层

建议新增 `NovelToScriptService`，职责只做三件事：

1. 标准化输入
   - 接收文本或导入结果
   - 按章节切分
   - 统计字数

2. 生成转换 Prompt
   - 根据配置拼接系统提示与用户提示
   - 控制输出格式

3. 清洗输出结果
   - 尝试解析结构化结果
   - 失败时至少保留完整文本结果

建议不要直接把全部逻辑堆进 route。

### 6.2 路由建议

建议新增接口：

1. `POST /api/novel-to-script/import`
   - 文件导入
   - 返回章节结构和原文文本

2. `POST /api/novel-to-script/convert`
   - 执行转换
   - 接收源文本/源章节和配置
   - 返回结构化剧本结果

3. `GET /api/novel-to-script/state`
   - 读取项目当前草稿

4. `POST /api/novel-to-script/state`
   - 保存当前草稿

5. `POST /api/novel-to-script/export?format=txt|md|docx`
   - 导出剧本结果

### 6.3 与现有能力的复用点

可直接复用：
- `NovelImportService.parse_novel_file()`：导入和章节拆分
- `project-state` 接口模式：项目内状态持久化
- `continuous_write.py` 导出文本和 DOCX 的方法
- `short_story.py` 的模型解析、接口超时控制、错误提示模式

## 7. Prompt 设计建议

### 7.1 第一版不要一步到位做“工业级剧本”

建议先做“稳定转换”，目标是：
- 把叙事文本改写成场景化表达
- 提取角色
- 把旁白改写为动作说明
- 把关键冲突改写为对白与场景推进
- 尽量贴近你给出的“场景块台本格式”，而不是输出成普通摘要

### 7.2 推荐两段式转换

建议不要让模型直接从超长全文一次输出最终剧本。

建议流程：

1. 分析阶段
   - 提取标题、人物、章节梗概、主要冲突
   - 给每章生成若干“场景卡”

2. 转写阶段
   - 根据场景卡生成正式台本文本

原因：
- 更稳
- 更容易续转失败重试
- 更容易以后支持“只重转第 N 章”

### 7.3 建议的结构化输出格式

建议模型优先输出 JSON 包裹的结构化结果，再由前端渲染成人类可读台本。

但这里要明确区分：
- `内部格式`：结构化 JSON，便于保存、局部重转、导出
- `最终展示格式`：必须尽量贴近你给出的场景块文本样式

建议格式：

```json
{
  "scenes": [
    {
      "scene_number": 1,
      "scene_label": "场景一",
      "heading": "青云门后山茅厕 - 暴雨夜",
      "characters_text": "夜天（青云门杂役弟子，二十岁左右，衣衫破旧，面带疲惫）",
      "environment_text": "简陋茅厕，木门破旧，被风吹得“吱呀”作响。",
      "beats": [
        {
          "type": "action_narration",
          "label": "动作/旁白",
          "text": "夜天蹲在茅厕角落，手里攥着一个破瓢。"
        },
        {
          "type": "character_line",
          "speaker": "夜天",
          "qualifier": "内心独白，语气绝望，低声呢喃",
          "text": "我真的只想躺平啊……"
        },
        {
          "type": "flashback_line",
          "label": "闪回片段",
          "qualifier": "快速切换，语气凶狠",
          "text": "刘管事指着夜天的鼻子，唾沫星子飞溅。"
        },
        {

          "text": "一道黑色闪电划破天际，“咔嚓”一声巨响。"

```

如果模型不稳定：
- 保留兜底文本输出
- 后端只要能返回 `formatted_text` 或 `full_text`，前端就能先展示

### 7.4 最终文本模板建议

为了让模型更稳定贴近你要的格式，Prompt 里建议明确要求最终文本严格遵守这个模板：

```text
【场景X：地点 - 时间/天气】
人物：角色A（简介）、角色B（简介）
环境：场景环境描述
动作/旁白：该场景开场动作与状态描述
角色名（内心独白，语气XXX）：内容
角色名（动作）：内容
角色名（对白，语气XXX）：内容
闪回片段（快速切换，语气XXX）：内容
动作/音效：内容
```

并加上约束：
- 不要输出 Markdown 标题、列表符号、编号列表
- 不要输出 JSON 解释文字
- 每个标签单独占一行
- 场景之间空一行
- 场景编号使用中文数字
- 优先保留原文人物语气和网文化表达

## 8. 前端拆分建议

建议新增文件：

- `novel_agent/web/static/app-novel-to-script.js`
- `novel_agent/web/static/novel-to-script/novel-to-script-state.js`
- `novel_agent/web/static/novel-to-script/novel-to-script-api.js`
- `novel_agent/web/static/novel-to-script/novel-to-script-render.js`
- `novel_agent/web/static/novel-to-script/novel-to-script-events.js`

原因：
- 当前 `short-story` 已经证明这种拆分方式可维护
- 这个新模块也明显是“独立业务域”
- 不建议把逻辑塞进 `app-nav.js` 或 `app-core.js`

### 8.1 资源栏改动

需要修改：
- `index.html`：新增一个 `data-module="novel-to-script"` 的按钮
- `app-core.js`：补 `switchModule('novel-to-script')` 分支
- `app-nav.js`：补 `renderNavPanel('novel-to-script')` 分支

### 8.2 页面表现建议

建议直接做成双栏：
- 左栏：输入、导入、转换参数
- 右栏：结构化剧本预览

如果结果较长：
- 右栏上方增加“人物表 / 场景目录 / 剧本正文”三个切换 Tab

但结合你给的截图，第一版展示层建议增加一个默认视图：
- `纯文本台本视图`

原因：
- 你要的最终格式本质上是连续文本台本
- 截图也说明这种格式在视觉上更接近“高可读纯文本”
- 所以结构化卡片视图可以有，但默认应展示为可复制、可编辑、按段落排版的纯文本结果

## 9. 测试建议

至少补三层测试。

### 9.1 服务层测试

新增：
- 文本输入能正常切章
- 导入 docx 能提取内容
- 大段无章节文本能降级为单章
- 结构化结果解析失败时能回退到纯文本结果
- 能正确把 `beats` 渲染回 `formatted_text`
- 能保留 `动作/旁白`、`动作/音效`、`闪回片段` 这类非角色标签

### 9.2 路由测试

新增：
- `import` 支持 `txt/md/docx`
- 非法格式报错
- 空文件报错
- `convert` 在未配置 API 时给出清晰错误
- `state` 能成功保存和读取
- `convert` 返回结果中包含 `formatted_text`

### 9.3 前端测试

参考现有 `frontend-tests` 风格，至少覆盖：
- 模块切换后能渲染工作台
- 文件导入成功后状态落库
- 转换按钮 loading 状态正确
- 导出按钮在无结果时禁用
- 纯文本台本视图能正确展示 `【场景X：...】` 和标签行

## 10. 推荐开发顺序

建议按 4 个阶段做。

### 阶段一：壳子和导入

目标：
- 左侧出现模块入口
- 页面能打开
- 支持粘贴文本与导入文件
- 保存项目内草稿

### 阶段二：转换 MVP

目标：
- 完成 `convert` 接口
- 输出剧本正文
- 支持复制和 TXT 导出

### 阶段三：结构化结果

目标：
- 输出人物表
- 输出场景目录
- 支持按场景浏览
- 支持 DOCX 导出

### 阶段四：增强能力

目标：
- 按章节局部重转
- 对白强化
- 网文短剧风格模板
- 可选“分镜倾向”输出

## 11. 风险与注意点

### 11.1 最大风险不是导入，而是模型输出稳定性

导入层现在已经比较成熟，真正难点是：
- 长文本上下文过长
- 模型输出不严格按格式
- 场景切分不稳定
- 对白容易“AI 腔”

所以第一版不要承诺：
- 影视工业标准格式
- 百分百可拍摄剧本
- 全自动分镜

### 11.2 不建议第一版直接转换超长整本小说

建议在产品文案中明确：
- 第一版更适合短篇、单章、中短篇分段转换
- 长篇建议按章节或分卷转换

### 11.3 不要复用协作创作的 `outline`

因为“小说大纲”与“剧本场景列表”不是同一种结构。

如果强行复用：
- 会让数据含义混乱
- 会影响现有协作创作模块
- 后期很难维护

## 12. 验收标准

MVP 完成后，至少满足以下标准：

1. 左侧存在独立入口，点击后能正常进入模块。
2. 支持粘贴文本和导入 `txt/md/docx`。
3. 能把输入文本转换为你示例这种“场景块台本格式”。
4. 页面能展示完整台本文本，且格式包含 `场景标题 / 人物 / 环境 / 动作/旁白 / 角色行 / 动作/音效` 等核心元素。
5. 支持复制结果与导出 `txt`，最好同时支持 `docx`。
6. 项目切换后，转换草稿不串项目。
7. 异常时能明确提示“格式错误 / 文件为空 / API 未配置 / 转换失败”。

## 13. 最终建议

从当前项目结构看，这个需求最合理的做法不是“在现有某个模块里塞一个按钮”，而是：

1. 新增一个独立左侧模块 `novel-to-script`
2. 复用现有 `NovelImportService` 做导入解析
3. 参考 `short-story` 的模块化前端拆分
4. 参考 `continuous_write` 的导入与导出能力
5. 用项目级独立状态保存转换草稿和结果

这样改动面清晰，风险可控，也不会污染现有小说创作流程。
