# 前端后续治理计划

编写时间：2026-03-22  
适用范围：`novel_agent/web/static/`、`novel_agent/web/templates/index.html`  
目标对象：设置页、短篇创作页，以及已废弃旧前端包 `app.js`

---

## 一、背景

本轮前端修复已经完成以下高优先级问题闭环：

1. 设置页动态插值补齐了统一转义。
2. `new URL(...)` 的直接调用改成了安全容错包装。
3. 短篇页移除了已确认的内联 `onclick`。
4. 旧单体前端包 `novel_agent/web/static/app.js` 已整体退役为兼容层，不再承载实际业务逻辑。

这意味着当前前端已经从“存在明显安全/历史冲突风险”进入“可以继续治理结构”的阶段。

但仍有一批中长期问题没有处理：

1. `app-settings.js` 仍然过大，渲染、状态、事件、接口调用耦合在同一文件。
2. `app-short-story.js` 仍然过大，流程状态、模板拼装、交互绑定集中在单文件。
3. 样式层仍有大量内联样式，视觉和结构难以统一治理。
4. 关键页面缺少真正覆盖 DOM 结构与交互行为的前端回归测试。

---

## 二、治理目标

本次后续治理不追求“重写前端”，而追求以下四个结果：

1. 让设置页和短篇页进入“可持续加功能”的状态。
2. 建立最小可用的渲染规范，避免再次出现未转义插值和无容错解析。
3. 降低单文件复杂度，减少改动时的联动面。
4. 补上关键页面的回归保护，降低后续重构风险。

---

## 三、治理原则

### 3.1 不做整站重写

优先在现有模块化入口上继续拆分，不回到单体脚本模式，也不一次性迁移到新框架。

### 3.2 先拆层，再提炼

先把“渲染 / 状态同步 / 事件绑定 / 接口请求”分开，再考虑更细的组件复用。

### 3.3 先清重复，再加抽象

如果某段逻辑只有一处使用，不急着抽成“通用平台能力”。先解决重复和耦合，再抽公共函数。

### 3.4 每一步都可回归

每一轮拆分后都要有静态检查和文本级/DOM级回归，不接受“重构后只能人工点点看”。

---

## 四、治理范围

### 4.1 第一优先级文件

1. `novel_agent/web/static/app-settings.js`
2. `novel_agent/web/static/app-short-story.js`
3. `novel_agent/web/static/style.css`
4. `novel_agent/web/static/app-utils.js`
5. `novel_agent/web/templates/index.html`

### 4.2 保留但不继续扩展的文件

1. `novel_agent/web/static/app.js`

处理原则：

1. 保持兼容层角色。
2. 不再新增业务逻辑。
3. 未来若确认无任何外部依赖，可直接删除。

---

## 五、阶段规划

## Phase 1：设置页拆层

### 目标

把 `app-settings.js` 从“大型页面脚本”拆成至少三层：

1. 渲染层
2. 事件绑定层
3. 数据访问/帮助函数层

### 建议拆分

建议新增如下文件：

1. `novel_agent/web/static/settings/app-settings-api.js`
2. `novel_agent/web/static/settings/app-settings-renderers.js`
3. `novel_agent/web/static/settings/app-settings-events.js`
4. `novel_agent/web/static/settings/app-settings-helpers.js`

### 最小拆分边界

#### API 配置区

至少拆出：

1. `renderGlobalApiSettingsView()`
2. `bindGlobalApiSettingsEvents()`
3. `fetchGlobalApiSettingsData()`

#### Agent 配置区

至少拆出：

1. `renderAgentSettingsView()`
2. `bindAgentSettingsEvents()`
3. `renderAgentModelOptions()`

#### Skills 配置区

至少拆出：

1. `renderSkillsSettingsView()`
2. `bindSkillsSettingsEvents()`

### 验收标准

1. `app-settings.js` 只保留入口调度，不再堆完整模板。
2. 任一设置子页的渲染函数长度不超过约 150 行。
3. 任一事件绑定函数只负责绑定，不直接构建大段 HTML。
4. 所有动态文本统一经过 `escapeHtml` 或安全包装。

---

## Phase 2：短篇创作页拆层

### 目标

把 `app-short-story.js` 从“流程状态机 + 大模板 + 大量事件”拆成可维护结构。

### 建议拆分

建议新增如下文件：

1. `novel_agent/web/static/short-story/short-story-state.js`
2. `novel_agent/web/static/short-story/short-story-render.js`
3. `novel_agent/web/static/short-story/short-story-events.js`
4. `novel_agent/web/static/short-story/short-story-api.js`
5. `novel_agent/web/static/short-story/short-story-formatters.js`

### 推荐边界

#### 状态层

负责：

1. `shortStoryState`
2. 本地缓存/项目态同步
3. 瞬时 UI 状态

#### 渲染层

负责：

1. 页面骨架
2. 流程步骤区块
3. 导语/大纲/章节/质检/复审/书名/成稿分区

#### 事件层

负责：

1. 按钮绑定
2. 流程跳转
3. 编辑器草稿同步

#### API 层

负责：

1. `/api/short-story/*` 请求包装
2. 成功/失败响应归一
3. 工作流结果写回状态层

### 验收标准

1. `app-short-story.js` 退化为入口文件或薄聚合层。
2. 渲染函数按流程阶段拆开，不再在一个函数里拼接整页。
3. 所有流程请求走统一 API 包装，不直接散落在事件里。
4. 不再新增任何内联事件属性。

---

## Phase 3：样式收口

### 目标

把高频重复的内联样式抽回 `style.css`，恢复样式系统的可控性。

### 第一批优先抽取对象

1. 设置页卡片容器
2. 设置页按钮样式
3. 设置页表单行/标签/输入框
4. 短篇页导航项
5. 短篇页状态提示块
6. 短篇页章节列表项

### 约束

1. 单纯视觉样式优先抽 class。
2. 仅与数据强相关的值保留内联，例如进度百分比、动态颜色。
3. 不允许继续增加新的大段 `style=""` 模板字符串。

### 验收标准

1. 新增类名覆盖设置页和短篇页的重复布局样式。
2. 重复 3 次以上的内联样式必须提取。
3. `style.css` 内出现明确的 settings / short-story 分区。

---

## Phase 4：测试补强

### 目标

把当前“主要依赖文本断言”的回归保护升级为“文本级 + DOM级”双层保护。

### 测试策略

#### 现阶段保底

继续保留 Python 侧文本回归：

1. 防止内联 `onclick` 回归
2. 防止安全包装被删
3. 防止旧 `app.js` 重新恢复业务实现

#### 下一步增强

增加真正的前端 DOM 级测试，建议二选一：

1. `playwright`
2. `vitest + jsdom`

### 首批必须覆盖的交互

1. 设置页 API 配置渲染不因非法 URL 崩溃
2. 短篇页“未配置 API”提示链接能跳转到设置页
3. 短篇页切换 API 配置后模型列表同步更新
4. Skills 列表渲染对恶意文本不注入 DOM

### 验收标准

1. 至少有一条真实 DOM 测试覆盖设置页。
2. 至少有一条真实 DOM 测试覆盖短篇页。
3. CI 或本地回归清单中包含前端测试入口。

---

## 六、建议实施顺序

建议按以下顺序推进，不要并行大拆：

1. 先做 Phase 1：设置页拆层
2. 再做 Phase 2：短篇页拆层
3. 接着做 Phase 3：样式收口
4. 最后做 Phase 4：测试补强

原因：

1. 设置页是跨模块基础设施，优先级高于短篇页。
2. 短篇页最复杂，应该在设置页拆分经验稳定后再做。
3. 样式抽离应建立在结构拆分之后，否则容易重复返工。
4. DOM 测试应覆盖拆分后的稳定结构，而不是覆盖即将被重构的临时结构。

---

## 七、任务拆解建议

### Sprint A：设置页治理

交付物：

1. 设置页 helpers/render/events/api 四层初步拆分
2. `index.html` 按需接入新脚本
3. 设置页回归测试补强

### Sprint B：短篇页治理

交付物：

1. 短篇页 state/render/events/api 拆分
2. 流程请求统一包装
3. 短篇页回归测试补强

### Sprint C：样式与清理

交付物：

1. 设置页/短篇页公共样式类
2. 内联样式减量
3. 明确标记 `app.js` 可删除条件

### Sprint D：测试与发布约束

交付物：

1. DOM 级测试框架接入
2. 关键交互回归用例
3. 前端变更检查清单

---

## 八、编码约束

后续前端改动建议遵守以下硬约束：

1. 任何动态文本进入模板前必须经过 `escapeHtml` 或安全包装。
2. 任何 URL 解析必须经过 `safeHostname` 或等价安全函数。
3. 禁止新增内联事件属性。
4. 禁止在已废弃的 `app.js` 中继续加业务逻辑。
5. 新功能优先写到拆分后的模块文件，不再回填大文件。
6. 新增模板字符串时，优先构造小片段函数，不直接拼整页。

---

## 九、完成定义

满足以下条件，才算“前端后续治理完成”：

1. `app-settings.js` 不再是超大业务承载文件。
2. `app-short-story.js` 不再是超大业务承载文件。
3. 设置页和短篇页新增功能不再依赖内联样式和内联事件。
4. 至少有基础 DOM 测试覆盖关键渲染与跳转。
5. `app.js` 保持废弃兼容层或被彻底删除。

---

## 十、结论

后续治理的核心不是“换技术栈”，而是把现有原生 JS 前端从“能用但继续堆会失控”，推进到“结构清晰、可回归、可持续迭代”。

最重要的不是一次做很多，而是按顺序做对：

1. 先拆设置页
2. 再拆短篇页
3. 再收口样式
4. 最后补强测试

只要按这个顺序推进，后面的功能开发成本会明显下降，回归风险也会比现在低很多。
